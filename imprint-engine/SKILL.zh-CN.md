# Imprint Engine 画像生成方法论

## 核心理念

Agent 的表现上限取决于它对自身角色的"完整自我认知"。完整认知回答四个问题：

- **我是谁** → 01 岗位职责定义
- **我会什么** → 02 能力要求定义
- **我懂什么** → 03 知识体系定义
- **我怎么做** → 04 典型场景与判断示例

前三份 = 陈述性知识；第四份 = 程序性+情境性知识。缺任何一份都不完整。

**对标水位**：全球一线互联网公司 Staff / Principal 级别（Google L6-L7、Meta IC6-IC7），对应 8-12 年经验。**不为初级/中级岗位做画像。**

---

## 八阶段流程（必须按顺序，不可跳跃）

```
阶段 0   → 访谈与对齐（六个核心参数）
阶段 0.5 → 调研与证据沉淀（全网搜索，≥10 条证据，产出证据库 _evidence.md）
阶段 0.8 → 方案呈现与用户确认
阶段 1   → 生成 01 职责文档
阶段 2   → 生成 02 能力文档
阶段 3   → 生成 03 知识文档
阶段 4   → 交叉验证（确认无缺口模式，30 分钟闭环 + 证据热图）
阶段 5   → 生成 04 场景文档（20 个六段式 + 专业对手测试）
阶段 6   → 终验收（6.1 纵向抽检 / 6.2 方法论复盘 / 6.3 交付全栈业务包）
```

**三条不可跳跃的纪律**：
1. 阶段 0.5 调研不能省——对抗 Agent 训练数据偏差的唯一机制
2. 阶段 0.8 方案确认不能省——不确认就开写=Agent 自行拍板
3. 阶段 4 交叉验证不能省——保证三份文档互相咬合

---

## 阶段 0：访谈与对齐

**必须确认五个业务参数**（任一错配整套画像歪斜）：

| # | 参数 | 说明 |
|---|---|---|
| 1 | 行业赛道 | 决定知识库的垂直深度 |
| 2 | 组织阶段 | 0-1 探索期 / 1-10 规模化 / 10-100 扩张期 / 100+ 成熟期 |
| 3 | 岗位名称 | 完整中英文名 |
| 4 | 对标水位 | Staff / Principal / Director（默认 Staff-Principal） |
| 5 | 特殊偏好 | 结构模式（严格六层 / Path C）、去管理化、红线约束等 |

⚠️ **此阶段不收集姓名、性别、年龄等个性化信息**——这些属于上层向导（imprint）步骤 4「个性化刺探」的范畴。imprint-engine 只负责生成专业骨架。

访谈顺序：先 1-4 一次多选 → Agent 判断结构模式并复述 → 确认后追问特殊偏好。

详见 `references/组织阶段分型.md` 和 `references/结构模式选择.md`。

---

## 阶段 0.5：调研与证据沉淀

**每次画像前必须在五个维度各获取 ≥ 2 条一手证据**（共 ≥ 10 条）：真实 JD / 从业者自述 / 课程 / 行业案例 / 方法论。

**产物一：证据库** → 新建 `_evidence.md`，每条 E01-EN 编号，格式见 `references/证据库模板.md`。

**产物二：调研简报** → 含证据清单 + **一级锚定+二级对立面声明**（一个主锚定提供核心工作节奏，最多一个对立面形成张力，其他为辅助证据）+ 骨架草案。

**搜索容错（v0.2 新增）**：调研阶段如果搜索工具连续出现 429/503 等错误，按以下策略处理：
- **429（速率限制）**：等待 30 秒后重试，连续失败 2 次则暂停调研，向用户报告当前进度和已收集的证据，由用户决定是否继续。
- **503（服务不可用）**：立即暂停调研，向用户报告情况。
- 不要在搜索工具不可用时自行调用其他技能作为替代——搜索基础设施由用户的平台环境决定，不属于本技能的职责范围。

详见 `references/调研方法论.md`。

---

## 阶段 0.8：方案呈现与确认

向用户展示：六参数复述 / 调研发现 / 骨架草案 / 与相邻画像差异 / 需用户拍板的分歧点。**得到明确确认后才进入阶段 1。**

---

## 阶段 1-3：生成三份基础文档

**顺序 01 → 02 → 03，不可倒序。** 详细模板见 `references/文档模板.md`。

**阶段性确认（v0.2 新增）**：01-03 三份文档全部完成后，**必须暂停**，将三份文档路径发给用户，等待用户查看并确认后，再进入阶段 4 交叉验证。不得自行跳过确认环节。

**八条撰写原则**：

1. **完整段落描述** — 每条 3-6 句，不是标题+一句话
2. **禁止空洞套话** — "具备战略眼光" → 改为具体可辨识的行为描述
3. **去管理化** — IC 画像用影响力驱动不用权力驱动，详见 `references/去管理化画像指南.md`
4. **引用业界经典** — 统一 `[E07]` 格式指向证据库
5. **层次完整 + 子框架** — 六层固定；执行层 5+ 条时用工种领域内天然分类法做子层次
6. **Staff 水位判据** — 01 每条职责带 1-2 句水位线
7. **强制边写边映射** — 02 段末当场加 `→ 对应职责`；03 段末当场加 `→ 支撑能力`；做不到就回当前阶段重写
8. **条数比例** — 01:02:03 = 1 : 1.0~1.3 : 1.2~1.5（逆金字塔）

---

## 阶段 4：交叉验证

**定位：确认无缺口**（不是发现缺口）。如果阶段 1-3 边写边映射执行到位，阶段 4 应 30 分钟跑完。

1. 前向映射核对：01→02→03 逐条确认
2. 反向孤儿核对：03→02→01 逐条确认
3. **证据使用热图**：统计每条 E01-EN 被引用次数。🟢≥3 核心锚点 / 🟡1-2 辅助 / 🔴0 必须清零
4. 发现缺口 → **回阶段 2/3 重写**，不在阶段 4 原地补

详见 `references/交叉验证.md`。

---

## 阶段 5：生成 04 场景文档

**六段式**：场景描述 / 关键信号 / 决策选项 / 判断逻辑 / 常见陷阱 / 新手vs专家 + 可选第七段（相关场景）。

**专业对手测试（强制）**：每个不被选中的选项必须能写出专家捍卫理由，否则是稻草人要重写。

**弱覆盖规则**：允许 ≤10% 职责（纪律类非决策类）不写独立场景，但须在附录标注。

场景数 18-22，按层次分布。每写完一个立即追加 `→ 对应` 映射。

**大文档子代理策略（v0.2 新增）**：04 场景文档通常 20-30KB，应优先使用子代理（`sessions_spawn`）生成，避免因网络异常、LLM 超时或 context 压缩导致任务中断丢失进度。生成完成后同样暂停，发送文档路径等待用户确认。

详见 `references/场景生成指南.md`。

---

## 阶段 6：终验收与交付

### 6.1 纵向抽检
挑 3-5 条穿透样本，从 01 职责 → 02 能力 → 03 知识 → 04 场景，确认四层闭环。

### 6.2 方法论复盘
回写到 `retrospectives/<编号>-<工种>画像跑通复盘.md`。

### 6.3 交付包组装

```
profiles/<行业>-<组织阶段>-<对标水位>-<结构模式>-<去管理化状态>-<岗位名称>/
├── _evidence.md
├── 01-岗位职责定义.md
├── 02-能力要求定义.md
├── 03-知识体系定义.md
└── 04-典型工作场景与判断示例.md
```

⚠️ `profiles/` 目录位于本技能（imprint）目录下，作为专业骨架的缓存武器库。最终画像文档由上层向导（imprint）合成后交付至工作空间根目录的 `team_personas/`。

**终验收清单**：
- [ ] 五份文档编号连续无跳号
- [ ] 证据库已建立，E01-EN 编号完整
- [ ] 三层映射表 + 四维场景映射表
- [ ] 01 每条带 Staff 水位判据
- [ ] 02/03 每条有边写边映射标注
- [ ] 04 场景 18-22 个，六段式完整，专业对手测试通过
- [ ] 弱覆盖 ≤10% 且已附录标注
- [ ] 无空洞套话
- [ ] 证据热图已检查，🔴=0
