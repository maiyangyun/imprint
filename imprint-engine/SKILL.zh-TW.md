# Imprint Engine 畫像生成方法論

## 核心理念

Agent 的表現上限取決於它對自身角色的"完整自我認知"。完整認知回答四個問題：

- **我是誰** → 01 崗位職責定義
- **我會什麼** → 02 能力要求定義
- **我懂什麼** → 03 知識體系定義
- **我怎麼做** → 04 典型場景與判斷示例

前三份 = 陳述性知識；第四份 = 程式性+情境性知識。缺任何一份都不完整。

**對標水位**：全球一線網際網路公司 Staff / Principal 級別（Google L6-L7、Meta IC6-IC7），對應 8-12 年經驗。**不為初級/中級崗位做畫像。**

---

## 八階段流程（必須按順序，不可跳躍）

```
階段 0   → 訪談與對齊（六個核心引數）
階段 0.5 → 調研與證據沉澱（全網搜尋，≥10 條證據，產出證據庫 _evidence.md）
階段 0.8 → 方案呈現與使用者確認
階段 1   → 生成 01 職責文件
階段 2   → 生成 02 能力文件
階段 3   → 生成 03 知識文件
階段 4   → 交叉驗證（確認無缺口模式，30 分鐘閉環 + 證據熱圖）
階段 5   → 生成 04 場景文件（20 個六段式 + 專業對手測試）
階段 6   → 終驗收（6.1 縱向抽檢 / 6.2 方法論覆盤 / 6.3 交付全棧業務包）
```

**三條不可跳躍的紀律**：
1. 階段 0.5 調研不能省——對抗 Agent 訓練資料偏差的唯一機制
2. 階段 0.8 方案確認不能省——不確認就開寫=Agent 自行拍板
3. 階段 4 交叉驗證不能省——保證三份文件互相咬合

---

## 階段 0：訪談與對齊

**必須確認五個業務引數**（任一錯配整套畫像歪斜）：

| # | 引數 | 說明 |
|---|---|---|
| 1 | 行業賽道 | 決定知識庫的垂直深度 |
| 2 | 組織階段 | 0-1 探索期 / 1-10 規模化 / 10-100 擴張期 / 100+ 成熟期 |
| 3 | 崗位名稱 | 完整中英文名 |
| 4 | 對標水位 | Staff / Principal / Director（預設 Staff-Principal） |
| 5 | 特殊偏好 | 結構模式（嚴格六層 / Path C）、去管理化、紅線約束等 |

⚠️ **此階段不收集姓名、性別、年齡等個性化資訊**——這些屬於上層嚮導（imprint）步驟 4「個性化刺探」的範疇。imprint-engine 只負責生成專業骨架。

訪談順序：先 1-4 一次多選 → Agent 判斷結構模式並複述 → 確認後追問特殊偏好。

詳見 `references/組織階段分型.md` 和 `references/結構模式選擇.md`。

---

## 階段 0.5：調研與證據沉澱

**每次畫像前必須在五個維度各獲取 ≥ 2 條一手證據**（共 ≥ 10 條）：真實 JD / 從業者自述 / 課程 / 行業案例 / 方法論。

**產物一：證據庫** → 新建 `_evidence.md`，每條 E01-EN 編號，格式見 `references/證據庫模板.md`。

**產物二：調研簡報** → 含證據清單 + **一級錨定+二級對立面宣告**（一個主錨定提供核心工作節奏，最多一個對立面形成張力，其他為輔助證據）+ 骨架草案。

**搜尋容錯（v0.2 新增）**：調研階段如果搜尋工具連續出現 429/503 等錯誤，按以下策略處理：
- **429（速率限制）**：等待 30 秒後重試，連續失敗 2 次則暫停調研，向用戶報告當前進度和已收集的證據，由使用者決定是否繼續。
- **503（服務不可用）**：立即暫停調研，向用戶報告情況。
- 不要在搜尋工具不可用時自行呼叫其他技能作為替代——搜尋基礎設施由使用者的平臺環境決定，不屬於本技能的職責範圍。

詳見 `references/調研方法論.md`。

---

## 階段 0.8：方案呈現與確認

向用戶展示：六引數複述 / 調研發現 / 骨架草案 / 與相鄰畫像差異 / 需使用者拍板的分歧點。**得到明確確認後才進入階段 1。**

---

## 階段 1-3：生成三份基礎文件

**順序 01 → 02 → 03，不可倒序。** 詳細模板見 `references/文件模板.md`。

**階段性確認（v0.2 新增）**：01-03 三份文件全部完成後，**必須暫停**，將三份文件路徑發給使用者，等待使用者檢視並確認後，再進入階段 4 交叉驗證。不得自行跳過確認環節。

**八條撰寫原則**：

1. **完整段落描述** — 每條 3-6 句，不是標題+一句話
2. **禁止空洞套話** — "具備戰略眼光" → 改為具體可辨識的行為描述
3. **去管理化** — IC 畫像用影響力驅動不用權力驅動，詳見 `references/去管理化畫像指南.md`
4. **引用業界經典** — 統一 `[E07]` 格式指向證據庫
5. **層次完整 + 子框架** — 六層固定；執行層 5+ 條時用工種領域內天然分類法做子層次
6. **Staff 水位判據** — 01 每條職責帶 1-2 句水位線
7. **強制邊寫邊對映** — 02 段末當場加 `→ 對應職責`；03 段末當場加 `→ 支撐能力`；做不到就回當前階段重寫
8. **條數比例** — 01:02:03 = 1 : 1.0~1.3 : 1.2~1.5（逆金字塔）

---

## 階段 4：交叉驗證

**定位：確認無缺口**（不是發現缺口）。如果階段 1-3 邊寫邊對映執行到位，階段 4 應 30 分鐘跑完。

1. 前向對映核對：01→02→03 逐條確認
2. 反向孤兒核對：03→02→01 逐條確認
3. **證據使用熱圖**：統計每條 E01-EN 被引用次數。🟢≥3 核心錨點 / 🟡1-2 輔助 / 🔴0 必須清零
4. 發現缺口 → **回階段 2/3 重寫**，不在階段 4 原地補

詳見 `references/交叉驗證.md`。

---

## 階段 5：生成 04 場景文件

**六段式**：場景描述 / 關鍵訊號 / 決策選項 / 判斷邏輯 / 常見陷阱 / 新手vs專家 + 可選第七段（相關場景）。

**專業對手測試（強制）**：每個不被選中的選項必須能寫出專家捍衛理由，否則是稻草人要重寫。

**弱覆蓋規則**：允許 ≤10% 職責（紀律類非決策類）不寫獨立場景，但須在附錄標註。

場景數 18-22，按層次分佈。每寫完一個立即追加 `→ 對應` 對映。

**大文件分批生成策略（v0.5 更新）**：04 場景文件通常 20-30KB，必須分批生成以避免子代理 token 耗盡：
- 將 18-22 個場景按層次分為 3-4 批，每批 5-7 個場景
- 每批作為一個獨立的子代理任務（`sessions_spawn`），task prompt 內聯：場景列表、對應的職責/能力對映、證據摘要
- 單個子代理 task prompt 控制在 8KB 以內
- 每批子代理寫入臨時檔案（如 `04-part-1.md`），全部完成後由主會話合併為最終的 `04-典型工作場景與判斷示例.md`
- 生成完成後同樣暫停，傳送文件路徑等待使用者確認

詳見 `references/場景生成指南.md`。

---

## 階段 6：終驗收與交付

### 6.1 縱向抽檢
挑 3-5 條穿透樣本，從 01 職責 → 02 能力 → 03 知識 → 04 場景，確認四層閉環。

### 6.2 方法論覆盤
回寫到 `retrospectives/<編號>-<工種>畫像跑通覆盤.md`。

### 6.3 交付包組裝

```
profiles/<行業>-<組織階段>-<對標水位>-<結構模式>-<去管理化狀態>-<崗位名稱>/
├── _evidence.md
├── 01-崗位職責定義.md
├── 02-能力要求定義.md
├── 03-知識體系定義.md
└── 04-典型工作場景與判斷示例.md
```

❗️ `profiles/` 目錄位於上層技能 imprint 的根目錄下（即 `skills/imprint/profiles/`），作為專業骨架的快取武器庫。注意不是 `skills/imprint/imprint-engine/profiles/`。最終畫像文件由上層嚮導（imprint）合成後交付至工作空間根目錄的 `soul-packages/`。

**終驗收清單**：
- [ ] 五份文件編號連續無跳號
- [ ] 證據庫已建立，E01-EN 編號完整
- [ ] 三層對映表 + 四維場景對映表
- [ ] 01 每條帶 Staff 水位判據
- [ ] 02/03 每條有邊寫邊對映標註
- [ ] 04 場景 18-22 個，六段式完整，專業對手測試透過
- [ ] 弱覆蓋 ≤10% 且已附錄標註
- [ ] 無空洞套話
- [ ] 證據熱圖已檢查，🔴=0
