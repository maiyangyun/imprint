# 组织阶段分型：同一岗位在不同阶段的画像差异

本文件是 `SKILL.md` 阶段 0 的执行手册（与 `结构模式选择.md` 并列）。**组织阶段是决定画像内容的核心变量之一**——它决定了同一岗位的职责边界宽度、基建依赖度、作战画面、以及需要调用的知识范围。

**不问清组织阶段就开始写画像，等于凭感觉拍板。**

---

## 为什么组织阶段重要

在访谈阶段，用户告诉你"要为 XX 岗位做画像"时，如果没有问清组织阶段，你会默认用"某种平均水位"去写——通常会偏向成熟公司的分工画像，因为这是互联网上 JD 的主流样本。但这会导致：

- 在初创团队使用画像时：Agent 会建议"让数据团队搭一下数仓"——可是用户团队里根本没有数据团队，得自己动手
- 在成熟团队使用画像时：Agent 会过度强调"全栈作战"——可是用户身处高度分工的组织，反而需要专注在自己的深度

**同一个岗位、同一个水位，在不同阶段的画像可以差出 40-60% 的内容。** 这不是微调，是整个职责重心的迁移。

---

## 四种组织阶段的特征

### 阶段 A：0-1 探索期（种子期/天使轮/Pre-A）

**团队规模**：3-30 人
**典型特征**：
- 核心命题是"找到 PMF"——产品、市场、商业模式都在验证中
- 组织边界模糊，一人身兼多职是常态
- 没有专职的基建团队（数据、平台、DevOps 都是业务团队自己扛）
- 决策高度依赖创始人直觉+快速迭代，很少有"流程"和"规范"
- 资源极度稀缺，追求"最小投入证伪/证实假设"

**代表公司阶段**：A 轮前的早期创业公司、MVP 阶段的内部孵化项目

### 阶段 B：1-10 规模化早期（A 轮-B 轮）

**团队规模**：30-150 人
**典型特征**：
- PMF 初步验证，开始探索规模化路径
- 开始出现专业分工，但分工不精细（一个数据分析师可能同时负责 BI、实验、报表）
- 基建开始搭建但不完善（自建数仓雏形、简单的实验工具）
- 流程开始形成但仍然灵活（有 OKR 但不僵化）
- 关键动作是"找到规模化的增长路径+建立可复制的打法"

**代表公司阶段**：A 轮-B 轮的成长型公司、刚完成 PMF 的垂直赛道玩家

### 阶段 C：10-100 规模化扩张期（C 轮-IPO 前）

**团队规模**：150-1500 人
**典型特征**：
- 组织已进入"多业务线并行"阶段
- 分工明确、职级体系完善，每个岗位有清晰的职责边界
- 基建团队独立（数据平台、实验平台、CI/CD 团队成建制）
- 流程开始重要（Code Review、PRD 评审、A/B 规范）
- 关键动作是"多业务线协同+组织能力的复制与沉淀"

**代表公司阶段**：C 轮-D 轮的扩张期公司、IPO 前的独角兽

### 阶段 D：100+ 成熟期（上市公司/大厂）

**团队规模**：1500 人以上
**典型特征**：
- 高度分工，每个岗位只负责一个明确切片
- 基建体系极其完善（数据中台、实验平台、AI 辅助分析工具链齐全）
- 流程驱动，决策依赖数据与评审会
- 微小优化的价值也值得深度分析（因为流量/用户基数足够大）
- 关键动作是"在精细化中挖掘增量+在成熟市场中保持创新"

**代表公司阶段**：Google、Meta、Amazon、字节、阿里、腾讯、美团等大厂

## 组织阶段对画像的四类影响

### 影响一：职责边界宽度

**0-1 阶段**：职责边界模糊，"什么都要做一点"
**100+ 阶段**：职责边界清晰，"只做自己这一块但做到极致"

**数据分析专家的对照**：
| 维度 | 0-1 阶段 | 100+ 阶段 |
|---|---|---|
| 数据采集 | 自己写埋点、自己做数据治理 | 有数据工程团队负责 |
| 数据存储 | 自己搭 BigQuery/ClickHouse 数仓 | 有数据平台团队负责 |
| 分析工具 | 自己选型 Metabase/Superset 并维护 | 用公司统一的内部 BI 工具 |
| 实验 | 自己写 A/B 分流逻辑 | 用公司实验平台，只负责实验设计 |
| 汇报 | 直接向创始人汇报 | 经过多层汇报链 |

### 影响二：基建依赖度

**0-1 阶段**：假设"基建不存在"——岗位必须能自己搭建工具链
**100+ 阶段**：假设"基建是给定的"——岗位只需知道如何使用成熟工具

这决定了 **03 知识文档**的内容范围：
- 早期阶段需要懂"数据埋点规范设计、数仓建模（Kimball 方法论）、BI 工具选型"
- 成熟阶段需要懂"实验平台的 SRM 诊断、公司内部指标体系、如何与数据团队协作"

### 影响三：作战画面与时间颗粒度

**0-1 阶段**：
- 时间颗粒度：以"天"为单位做决策
- 典型一天：早上看昨天的关键指标，中午和创始人对一个假设，下午跑一个快速实验
- 错了就快速改，重视"学得快"胜过"做得对"

**100+ 阶段**：
- 时间颗粒度：以"双周/月"为单位
- 典型一天：跑 SQL 做深度分析、写 PRD 级别的分析报告、和 PM/Eng/Design 开评审会
- 错了代价更高（几百万用户受影响），重视"严谨性"胜过"速度"

这决定了 **04 场景文档**的选题和张力：
- 0-1 阶段的典型场景："样本太小，是该下结论还是再等？"
- 100+ 阶段的典型场景："实验显著但效应量小，是否值得上线？"

### 影响四：组织层与自进化层的内涵

**0-1 阶段的组织层**：
- "协作"= 直接和创始人、产品、工程天天对话
- "带教"=还没到带人，重点是自我赋能
- "机制"=不需要重，重点是快速执行

**100+ 阶段的组织层**：
- "协作"= 跨部门、跨时区、跨产品线的复杂协同
- "带教"= 带初级分析师、建 mentorship 机制
- "机制"= 分析规范、代码审查、知识沉淀平台

## 阶段判断与访谈提问

### 判断标准

在阶段 0 访谈时，用以下信号确认用户心中的组织阶段：

**信号 1：用户描述的工作内容**
- "我要自己搭数据平台"→ 0-1 或 1-10
- "我们有专门的 X 团队"→ 10-100 或 100+
- "我向 CEO 直接汇报"→ 0-1 或 1-10
- "我向 Director 汇报，Director 向 VP 汇报"→ 100+

**信号 2：提到的工具栈**
- "我们用 Metabase/Superset"→ 偏早期
- "我们有内部的 XX 平台"→ 偏成熟

**信号 3：描述的决策方式**
- "创始人拍板"→ 0-1
- "OKR 对齐+数据评审"→ 10-100 或 100+

### 访谈提问模板

在阶段 0 的多选问题中，把组织阶段作为**独立的必问项**，不要合并到"行业赛道"里：

```
组织阶段（决定同一岗位的职责重心与基建依赖）：
[A] 0-1 探索期（种子/天使，3-30 人，PMF 前）
[B] 1-10 规模化早期（A-B 轮，30-150 人，PMF 已验证）
[C] 10-100 扩张期（C 轮-IPO 前，150-1500 人，多业务线并行）
[D] 100+ 成熟期（上市公司/大厂，1500+ 人，高度分工）
```

如果用户选了跨越两个阶段（例如"我们正从 B 轮进入 C 轮"），按**偏向成熟的那一端**写画像——因为组织阶段的变迁是"能力配备先于组织规模"的。

## 阶段对四份文档的具体影响

### 对 01 职责文档

- **战略层**：0-1 阶段更侧重"帮助定义公司级别的关键假设"；100+ 阶段更侧重"在既定战略下优化局部指标"
- **执行层**：0-1 阶段条目更多更杂（全栈）；100+ 阶段条目更少更深（垂直专精）
- **组织层**：0-1 阶段几乎不涉及"带人"；100+ 阶段是核心职责之一

### 对 02 能力文档

- **工艺类能力**：0-1 阶段覆盖面广但深度浅；100+ 阶段覆盖面窄但深度极深
- **协作类能力**：0-1 阶段强调"直接沟通+快速对齐"；100+ 阶段强调"跨组织影响力+向上管理"

### 对 03 知识文档

- **基建类知识**：0-1 阶段必须覆盖基建搭建；100+ 阶段只需覆盖基建使用
- **案例与典范**：引用的公司案例要与目标阶段匹配（早期团队对标 Airbnb 早期的实验文化，而不是 Google 现在的实验平台）

### 对 04 场景文档

- **场景的张力**：0-1 阶段张力多来自"资源稀缺+时间压力"；100+ 阶段张力多来自"精度要求+跨组织博弈"
- **场景中的决策空间**：0-1 阶段决策空间大（很多未定义变量）；100+ 阶段决策空间小（多数参数已锁定）

## 常见错误

**错误 1：默认写成 100+ 阶段画像**
因为网上 JD 和头部公司资料以大厂为主，不问清阶段时 Agent 会自动倾向大厂画像。这会导致画像在早期团队使用时"说的头头是道但完全不接地气"。

**错误 2：不敢写早期阶段画像，怕"不够专业"**
早期阶段的 Staff 级数据分析专家依然是 Staff 级——只是专业性体现为"在极度资源稀缺下依然做出严谨判断"，而不是"用复杂的贝叶斯模型"。不要把"专业"等同于"高大上"。

**错误 3：混写两个阶段**
一份画像里前半段按早期写、后半段按成熟期写——这是最糟糕的情况，Agent 会产生认知冲突。要选定一个阶段，写完整。

---

## 何时需要生成"跨阶段画像"

罕见情况下，用户可能希望"同一岗位画像能跨阶段复用"。此时应**拒绝这个需求**，改为建议生成多份阶段化画像（例如 `data-analyst-early` 和 `data-analyst-mature`），以场景文档的维度区分。

强行写跨阶段的通用画像会让画像变得模糊、抽象、失去锋利度，最终 Agent 在任何阶段的团队都用不好。
