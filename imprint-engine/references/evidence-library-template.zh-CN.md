# 证据库模板：`_evidence.md` 的结构与引用规则

本文件是 `SKILL.md` 阶段 0.5 的执行手册补充。它规定了 `profiles/<role-name>/_evidence.md` 证据库文件的字段、编号规则、在 01/02/03/04 里的引用格式，以及阶段 4 的"证据使用热图"检查方法。

---

## 为什么需要一个独立的证据库

在旧版方法论里，阶段 0.5 的调研产物只以"调研简报"的形式在对话中呈现，**没有沉淀为文件**。带来的副作用：

- 写 01/02/03/04 时，引用格式各自为政——有的写"参考 Reforge 课程"、有的写"某位 Airbnb 前数据科学家"、有的直接不引用
- 读者（包括 Agent 自己和用户）无法顺着引用去回查原始调研
- 没有办法量化"某条证据在画像里到底被用了几次"——调研效率是盲的
- 同一条证据在不同文档里的表述不一致，像是"多个来源的平均拼贴"

**独立证据库解决这四个问题**：统一编号 + 统一引用格式 + 可追溯 + 可量化。

---

## 证据库文件位置与命名

每个画像都必须有一个专属的证据库文件：

```
profiles/<role-name>/_evidence.md
```

文件名以下划线开头，表示它是"这个画像的附件/元数据"而非画像正文。

---

## 证据库的结构

```markdown
# <角色中文名> | 证据库（_evidence.md）

> **画像**：<角色中文名 / 英文名>
> **调研日期**：<YYYY-MM-DD>
> **证据总数**：<N>
> **一级锚定**：<E0X，一级锚定证据的编号>
> **二级对立面**：<E0Y，二级对立面证据的编号>
> **使用热图最后更新**：<YYYY-MM-DD>

---

## 证据清单

### E01 <证据标题>

- **来源类型**：真实 JD / 从业者自述 / 课程 / 行业案例 / 方法论 / 学术论文
- **来源链接**：<URL 或"无链接，来自 XX 书籍第 YY 页">
- **作者/机构**：<人名或公司>
- **调研时间**：<YYYY-MM-DD>
- **关键摘要**：<一段 2-5 句话，用自己的话重述这条证据的核心内容。不要直接复制原文，要抓住"对本画像有用的那个点"。>
- **对画像的影响**：<具体说明它塑造了 01/02/03/04 的哪一部分。例：影响了 01 第三层的 3C 子框架划分 / 为 02-2.4 能力提供了"先写证伪条件"的专家判断依据 / 成为 04 场景 2.1 的情境原型>
- **引用次数**（阶段 4 后更新）：01×_ / 02×_ / 03×_ / 04×_ = 总 _ 次
- **是否一级锚定 / 二级对立面**：✅ 一级锚定 / ✅ 二级对立面 / —

---

### E02 <证据标题>

（同上格式）

---

### ...（E03、E04、E05……直到 EN）
```

## 编号规则

- 证据编号从 **E01** 开始，顺序递增：E01、E02、E03、……、E10、E11、……、EN
- 所有证据使用**两位数字**（E01 而非 E1）——这样在后续引用中 E07 和 E17 的 `[E07]` 在视觉上对齐
- 编号一旦分配就不再变动；如果某条证据在后期被删除，**保留编号占位并在条目里写明"已作废"**，下面解释为什么
- 不允许复用编号：删除的编号空出来即可，新增证据用下一个未使用的编号

**为什么删除证据时要保留编号占位**：

假设 E07 是 "Google Good Data Analysis 指南"，后来发现它对画像贡献很小，要删除。如果直接删掉、把后面的 E08 改名为 E07，那么 01/02/03/04 里所有 `[E07]` 引用都会指向错误的证据。正确做法是：

```markdown
### E07 ~~Google Good Data Analysis 指南~~（已作废）

- **作废原因**：阶段 4 热图显示引用次数为 0；经复查，本证据对本画像的贡献已被 E03 和 E12 覆盖，无独立增量价值
- **作废日期**：<YYYY-MM-DD>
```

这样既不破坏已有引用，也保留了决策的审计痕迹。

---

## 引用格式（在 01/02/03/04 里如何用）

**统一使用 `[E07]` 格式**，内嵌在段落文字中：

> Amazon 的 "Working Backwards" [E03]、Airbnb 早期的 "Belong Anywhere" [E11] 都是这一职责的典范。

不要用 "[Amazon, Working Backwards]" 这样的零散引用，也不要用脚注编号 `[1]`——都不利于跨文档的证据热图统计。

**引用位置建议**：

- 01 职责文档：主要出现在"为什么这个职责对本岗位重要"的那一句、或"引用头部公司做法"的那一句
- 02 能力文档：主要出现在"专家级与初级水位差异"的那一句、或"典型训练路径"的那一句
- 03 知识文档：主要出现在"这个概念的提出者/经典出处"的那一句
- 04 场景文档：主要出现在判断逻辑段（引用某位专家的原话或方法论）、常见陷阱段（引用某个著名失败案例）

## 一级锚定 + 二级对立面

阶段 0.5 的调研简报里已经要求声明"一级锚定 + 最多一个二级对立面"——证据库要把这两个位置**明确标记**出来。

**例子（数据分析专家画像）**：

```markdown
> **一级锚定**：E03（DoorDash 数据团队三角架构 / Jessica Lachs 在 Lenny's Podcast 的访谈）
> **二级对立面**：E11（Airbnb 数据科学团队二元结构 / Chetan Sharma 的内部方法论文章）
```

这两条证据应该在整个画像里被反复调用——理想情况下，它们的引用次数都应该达到 **5+ 次**，成为热图上的"核心锚点"（详见 `交叉验证.md` 的证据使用热图章节）。

**判据**：一级锚定提供"这个工种在一个真实公司里是怎么干活的"完整画面；二级对立面提供"在另一种组织形态/哲学下，同一个工种会怎么不同"的张力。两者结合就能让画像有"内在思辨性"，而不是多个来源的平均主义拼贴。

---

## 最低调研规模

每次生成画像前，证据库至少应包含 **10 条证据**，覆盖五个维度各 ≥ 2 条：

1. 真实 JD（≥ 2 条）
2. 从业者自述（≥ 2 条）
3. 课程/培训体系（≥ 2 条）
4. 行业案例（≥ 2 条）
5. 方法论/框架（≥ 2 条）

理想情况是 **15-25 条**。超过 30 条说明调研过度，应该精简到"只保留真正塑造画像的那些"。

## 阶段 4 的证据使用热图

见 `交叉验证.md` 的"证据库使用热图"章节。核心判据：

- 🟢 核心锚点（被引用 ≥ 3 次）：至少 3 条
- 🟡 辅助证据（被引用 1-2 次）：大部分证据在此区间
- 🔴 未被使用（被引用 0 次）：**必须清零**——要么在画像里补引用，要么从证据库里作废（保留编号占位 + 标注作废原因）

---

## 证据库的长期价值

一次画像做完后，证据库不是"一次性调研草稿"，而是这个画像的**长期参考资料**：

- 未来迭代画像时（季度审查、新场景扩展），可以基于证据库判断"需要补哪方面的新证据"
- 做相邻岗位的画像时（例如已有数据分析师，要做数据工程师），证据库可以被部分复用
- 当用户对画像里某条内容提出质疑时，可以顺着 `[E07]` 直接回查原始证据
- 审计画像的更新频率：长期没更新的证据库意味着画像本身已经滞后于行业

因此证据库应该和 01/02/03/04 一起保存在 `profiles/<role-name>/` 目录下，作为画像交付包的一部分。
