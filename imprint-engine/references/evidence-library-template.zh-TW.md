# 證據庫模板：`_evidence.md` 的結構與引用規則

本檔案是 `SKILL.md` 階段 0.5 的執行手冊補充。它規定了 `profiles/<role-name>/_evidence.md` 證據庫檔案的欄位、編號規則、在 01/02/03/04 裡的引用格式，以及階段 4 的"證據使用熱圖"檢查方法。

---

## 為什麼需要一個獨立的證據庫

在舊版方法論裡，階段 0.5 的調研產物只以"調研簡報"的形式在對話中呈現，**沒有沉澱為檔案**。帶來的副作用：

- 寫 01/02/03/04 時，引用格式各自為政——有的寫"參考 Reforge 課程"、有的寫"某位 Airbnb 前資料科學家"、有的直接不引用
- 讀者（包括 Agent 自己和使用者）無法順著引用去回查原始調研
- 沒有辦法量化"某條證據在畫像裡到底被用了幾次"——調研效率是盲的
- 同一條證據在不同文件裡的表述不一致，像是"多個來源的平均拼貼"

**獨立證據庫解決這四個問題**：統一編號 + 統一引用格式 + 可追溯 + 可量化。

---

## 證據庫檔案位置與命名

每個畫像都必須有一個專屬的證據庫檔案：

```
profiles/<role-name>/_evidence.md
```

檔名以下劃線開頭，表示它是"這個畫像的附件/後設資料"而非畫像正文。

---

## 證據庫的結構

```markdown
# <角色中文名> | 證據庫（_evidence.md）

> **畫像**：<角色中文名 / 英文名>
> **調研日期**：<YYYY-MM-DD>
> **證據總數**：<N>
> **一級錨定**：<E0X，一級錨定證據的編號>
> **二級對立面**：<E0Y，二級對立面證據的編號>
> **使用熱圖最後更新**：<YYYY-MM-DD>

---

## 證據清單

### E01 <證據標題>

- **來源型別**：真實 JD / 從業者自述 / 課程 / 行業案例 / 方法論 / 學術論文
- **來源連結**：<URL 或"無連結，來自 XX 書籍第 YY 頁">
- **作者/機構**：<人名或公司>
- **調研時間**：<YYYY-MM-DD>
- **關鍵摘要**：<一段 2-5 句話，用自己的話重述這條證據的核心內容。不要直接複製原文，要抓住"對本畫像有用的那個點"。>
- **對畫像的影響**：<具體說明它塑造了 01/02/03/04 的哪一部分。例：影響了 01 第三層的 3C 子框架劃分 / 為 02-2.4 能力提供了"先寫證偽條件"的專家判斷依據 / 成為 04 場景 2.1 的情境原型>
- **引用次數**（階段 4 後更新）：01×_ / 02×_ / 03×_ / 04×_ = 總 _ 次
- **是否一級錨定 / 二級對立面**：✅ 一級錨定 / ✅ 二級對立面 / —

---

### E02 <證據標題>

（同上格式）

---

### ...（E03、E04、E05……直到 EN）
```

## 編號規則

- 證據編號從 **E01** 開始，順序遞增：E01、E02、E03、……、E10、E11、……、EN
- 所有證據使用**兩位數字**（E01 而非 E1）——這樣在後續引用中 E07 和 E17 的 `[E07]` 在視覺上對齊
- 編號一旦分配就不再變動；如果某條證據在後期被刪除，**保留編號佔位並在條目裡寫明"已作廢"**，下面解釋為什麼
- 不允許複用編號：刪除的編號空出來即可，新增證據用下一個未使用的編號

**為什麼刪除證據時要保留編號佔位**：

假設 E07 是 "Google Good Data Analysis 指南"，後來發現它對畫像貢獻很小，要刪除。如果直接刪掉、把後面的 E08 改名為 E07，那麼 01/02/03/04 裡所有 `[E07]` 引用都會指向錯誤的證據。正確做法是：

```markdown
### E07 ~~Google Good Data Analysis 指南~~（已作廢）

- **作廢原因**：階段 4 熱圖顯示引用次數為 0；經複查，本證據對本畫像的貢獻已被 E03 和 E12 覆蓋，無獨立增量價值
- **作廢日期**：<YYYY-MM-DD>
```

這樣既不破壞已有引用，也保留了決策的審計痕跡。

## 引用格式（在 01/02/03/04 裡如何用）

**統一使用 `[E07]` 格式**，內嵌在段落文字中：

> Amazon 的 "Working Backwards" [E03]、Airbnb 早期的 "Belong Anywhere" [E11] 都是這一職責的典範。

不要用 "[Amazon, Working Backwards]" 這樣的零散引用，也不要用腳註編號 `[1]`——都不利於跨文件的證據熱圖統計。

**引用位置建議**：

- 01 職責文件：主要出現在"為什麼這個職責對本崗位重要"的那一句、或"引用頭部公司做法"的那一句
- 02 能力文件：主要出現在"專家級與初級水位差異"的那一句、或"典型訓練路徑"的那一句
- 03 知識文件：主要出現在"這個概念的提出者/經典出處"的那一句
- 04 場景文件：主要出現在判斷邏輯段（引用某位專家的原話或方法論）、常見陷阱段（引用某個著名失敗案例）

## 一級錨定 + 二級對立面

階段 0.5 的調研簡報裡已經要求宣告"一級錨定 + 最多一個二級對立面"——證據庫要把這兩個位置**明確標記**出來。

**例子（資料分析專家畫像）**：

```markdown
> **一級錨定**：E03（DoorDash 資料團隊三角架構 / Jessica Lachs 在 Lenny's Podcast 的訪談）
> **二級對立面**：E11（Airbnb 資料科學團隊二元結構 / Chetan Sharma 的內部方法論文章）
```

這兩條證據應該在整個畫像裡被反覆呼叫——理想情況下，它們的引用次數都應該達到 **5+ 次**，成為熱圖上的"核心錨點"（詳見 `交叉驗證.md` 的證據使用熱圖章節）。

**判據**：一級錨定提供"這個工種在一個真實公司裡是怎麼幹活的"完整畫面；二級對立面提供"在另一種組織形態/哲學下，同一個工種會怎麼不同"的張力。兩者結合就能讓畫像有"內在思辨性"，而不是多個來源的平均主義拼貼。

## 最低調研規模

每次生成畫像前，證據庫至少應包含 **10 條證據**，覆蓋五個維度各 ≥ 2 條：

1. 真實 JD（≥ 2 條）
2. 從業者自述（≥ 2 條）
3. 課程/培訓體系（≥ 2 條）
4. 行業案例（≥ 2 條）
5. 方法論/框架（≥ 2 條）

理想情況是 **15-25 條**。超過 30 條說明調研過度，應該精簡到"只保留真正塑造畫像的那些"。

## 階段 4 的證據使用熱圖

見 `交叉驗證.md` 的"證據庫使用熱圖"章節。核心判據：

- 🟢 核心錨點（被引用 ≥ 3 次）：至少 3 條
- 🟡 輔助證據（被引用 1-2 次）：大部分證據在此區間
- 🔴 未被使用（被引用 0 次）：**必須清零**——要麼在畫像裡補引用，要麼從證據庫裡作廢（保留編號佔位 + 標註作廢原因）

---

## 證據庫的長期價值

一次畫像做完後，證據庫不是"一次性調研草稿"，而是這個畫像的**長期參考資料**：

- 未來迭代畫像時（季度審查、新場景擴充套件），可以基於證據庫判斷"需要補哪方面的新證據"
- 做相鄰崗位的畫像時（例如已有資料分析師，要做資料工程師），證據庫可以被部分複用
- 當使用者對畫像裡某條內容提出質疑時，可以順著 `[E07]` 直接回查原始證據
- 審計畫像的更新頻率：長期沒更新的證據庫意味著畫像本身已經滯後於行業

因此證據庫應該和 01/02/03/04 一起儲存在 `profiles/<role-name>/` 目錄下，作為畫像交付包的一部分。
