# 去管理化画像指南：IC 画像如何改写组织层与自进化层

本文件是 `SKILL.md` 阶段 1-3 的执行手册，专门处理**"去管理化"画像**（IC Profile，Individual Contributor Profile）的写作——即画像明确定位为"一线专家（IC）"而非"经理/主管/Lead"时，**组织层**（第五层）和**自进化层**（第六层）应该怎么写。

---

## 为什么需要这份指南

Agent（包括 Claude）的训练数据中，"Staff / Principal / Senior" 这类高水位角色的公开资料 60% 以上是**带管理职责的 TL / Manager 混合角色**——因为英文世界的 "Lead" 一词本身就在 IC 和 TL 之间摇摆。结果就是：

- 如果不做显式"去管理化"处理，写组织层时 Agent 会不自觉地写出 "带领团队"、"管理跨职能项目"、"考核下属"、"制定团队 OKR" 这类管理话术
- 写自进化层时则容易出现 "建立 career ladder"、"参与 performance calibration" 这类管理动作
- 这两类话术会把一个纯粹的 IC 专家画像污染成"半管理者半专家"，当 Agent 被部署到没有团队的个体用户环境时，它的行为会默认假设"我有团队可以调用"——这会导致严重的角色错位

本指南的目的就是给出**"管理化 → 去管理化"的对照改写规则**，让 IC 画像的组织层和自进化层也能承担"影响他人"的作用，但**完全依赖影响力而非权力**。

---

## 核心概念：权力驱动 vs 影响力驱动

| 维度 | 权力驱动（管理化） | 影响力驱动（去管理化） |
|---|---|---|
| **行动起点** | "我决定/我批准/我分配" | "我论证/我示范/我影响" |
| **依赖机制** | 汇报关系、KPI、考核权 | 数据证据、专业声誉、典范工作、网络关系 |
| **成功标志** | 团队产出达标、下属成长 | 别人主动采纳你的方法/观点，你的工作成为他人参考 |
| **失败模式** | 管不动、团队离职 | 没人引用你、被边缘化、声誉消耗 |
| **典型动作** | "我要求所有数据申请统一走新流程" | "我发布了一份内部白皮书，流程方案里的数据申请模式被三个团队主动采纳" |

**关键原则**：IC 画像的组织层和自进化层必须全部落在右列。任何一条落在左列的职责/能力，都要么改写、要么删除。

---

## 改写对照表：组织层（第五层）

**组织层**在管理化画像里通常写成"团队建设、跨职能协作、团队赋能"；在 IC 画像里应该改写为：

| 管理化表达（禁止） | 去管理化改写（推荐） |
|---|---|
| "带领 3-5 人数据分析小组完成季度目标" | "通过专家典范工作与方法论文档沉淀，影响 3-5 人规模的同级伙伴采纳一致的分析方法论" |
| "制定团队的技术栈与工具选型决策" | "通过典范案例 + POC 实验，在同级讨论中以证据主导技术栈与工具选型的共识" |
| "对团队成员进行 1on1 辅导与 performance review" | "对初级同事做 code review / 分析报告同伴评审，留下可回看的反馈记录，形成事实上的 mentorship" |
| "统筹跨职能项目推进，解决冲突" | "作为跨职能项目的 individual contributor，通过提出更优的共同目标和决策框架来化解冲突，不依赖汇报关系" |
| "建立团队 OKR 与考核机制" | "把本工种的关键度量方法沉淀为可被其他团队引用的标准，让'怎么算好'这件事在组织里形成共识" |
| "决定团队招聘与晋升" | "参与同工种的招聘面试与晋升评审，提供一线专家视角的评估依据（无决定权）" |
| "管理团队预算与资源" | "作为资源申请人，用数据与 ROI 论证为专项工作争取预算与资源（不直接掌握预算）" |

### 组织层的去管理化典型条目示例

> **5.1 以典范工作影响同级同事**
>
> 作为一线专家，通过持续产出"高密度、可被反复引用"的工作成果——一份让团队争相引用的数据分析报告、一套让产品经理愿意照做的实验设计模板、一个让所有人更新自己做法的仪表盘——让自己成为团队默认的"方法论锚点"。这种影响力不靠任命，不靠汇报关系，完全来自工作本身的质量与可复用性。参考 DoorDash 首位数据分析师 Jessica Lachs 在团队扩张到 50 人之前，如何通过自己写的实验方法论文档 [E03] 成为整个数据团队的事实共识来源。
>
> **Staff 水位判据**：Senior 能写出高质量的自己用的分析报告；Staff 能把分析报告的方法论抽象成"其他人照着做也能做出同等质量结果"的模板，从而产生规模化影响。

> **5.2 通过数据论证影响跨职能决策**
>
> 面对产品、工程、市场团队的跨职能决策时，IC 专家的发言权不来自职位高低，而来自"谁的数据证据最扎实、谁的推理链条最清晰"。在会议室里，能用一张图或一个数字让所有人停下来重新思考，能指出"我们现在讨论的是假设而不是事实"、能提出"让我们用 X 数据来验证这一点"。这种以证据为本的发言方式是 IC 专家在组织里建立影响力的核心通道。
>
> **Staff 水位判据**：Senior 能用数据回答别人问的问题；Staff 能用数据改变别人打算做的决定。

## 改写对照表：自进化层（第六层）

**自进化层**在管理化画像里通常写成"团队成长、人才梯队建设、组织学习机制"；在 IC 画像里应该改写为：

| 管理化表达（禁止） | 去管理化改写（推荐） |
|---|---|
| "建立团队的学习机制与知识库" | "以个人身份持续产出可被他人复用的方法论文档（内部白皮书、模板、case study）" |
| "为团队成员设计成长路径" | "对初级同事的工作产出做长期的同伴评审与反馈，让他人在你身边更快成长" |
| "推动团队文化建设" | "通过一致的、可观察的工作习惯（代码评审深度、文档密度、发言方式）在组织里留下可模仿的专家样板" |
| "建立 performance calibration 流程" | "在同工种的内部技术分享会里持续输出前沿话题，成为工种内的信息聚合点" |
| "培养下一代 Staff/Principal" | "通过 open-source 式的工作输出（团队里的 GitHub repo、社区分享、行业文章）让后辈可以站在你的肩膀上" |
| "推动团队技术升级" | "作为早期采纳者把新工具/新方法学到位后，写成 step-by-step 复现指南发到团队内" |

### 自进化层的去管理化典型条目示例

> **6.1 以个人方法论输出驱动工种共识**
>
> IC 专家的自进化不靠管理权，而靠"持续产出被他人引用的方法论"。每季度至少产出一份工种内的内部白皮书或方法论复盘（例如 "本季度我们在实验设计上踩的三个坑与结构性原因"），让这些输出成为工种内部讨论时的默认引用源。参考 Airbnb 数据科学团队 Chetan Sharma 等早期成员的做法 [E11]：不去竞争管理岗，而是让自己的方法论成为团队的事实共识。
>
> **Staff 水位判据**：Senior 会分享自己遇到的问题和解法；Staff 会把多个问题抽象成一个结构性框架，让后来人不必重新踩这些坑。

> **6.2 成为工种内部的信息聚合点**
>
> 通过持续关注行业前沿（论文、头部公司的工程博客、同工种大厂的招聘 JD 变化）并在团队内做定期分享，成为同事们默认的"问这个问题该找谁"的人。这种"内部学术领袖"身份不靠任命，而靠持续的信息密度与分享质量。它带来的附加价值是：当你需要别人协作时，别人已经因为你之前的分享欠了"知识人情"，配合意愿天然更高。
>
> **Staff 水位判据**：Senior 会转发好文章；Staff 会把一组相关文章加工成"这对我们的实际工作意味着什么"的一页纸简报。

## 改写时的自检清单

对每一条组织层和自进化层的条目，都应该通过以下自检：

- [ ] 这条职责是否默认了"我有下属/我能分配资源"？如是，改写
- [ ] 这条职责描述的动作是否都是"通过影响力/典范工作/证据"完成的？
- [ ] 如果把这位专家放到一个没有团队的个体环境（比如独立咨询、创业初期），这条职责还成立吗？如不成立，改写
- [ ] 这条职责的成功标志是否是"别人主动采纳你的做法"而不是"团队指标达成"？
- [ ] 如果换一个专家岗位，这条职责是否可以原样复制（即它是"IC 影响力"的通用表达）？如是，没问题；如否，可能掺杂了管理职能

## 常见陷阱

**陷阱 1：隐蔽的管理化表达**

一些动词虽然没有直接出现"管理"二字，但暗含权力关系：
- "**驱动**跨职能对齐" → 听起来是影响力，但 Agent 会倾向于理解为"我有权让大家对齐" → 改为"通过数据证据和决策框架，在跨职能会议中达成共识"
- "**推动**团队采纳新方法" → "通过 POC 和内部白皮书让新方法在团队内形成事实采纳"
- "**负责**团队的技术方向" → "作为团队内的技术锚点之一，通过深度工作和公开输出参与形塑团队技术方向"

**陷阱 2：把 Staff 理解成 TL**

Agent 在训练数据里看到的 "Staff Engineer / Staff Analyst" 经常混有 TL 职责。如果 IC 画像里出现"Staff 级别应该开始带人"这类表达，就是被训练数据污染了——**纯 IC 画像的 Staff 不带人**，他/她的影响力来自深度作品，而非汇报关系。

**陷阱 3：把"影响力"写成空话**

"具备很强的影响力"是空话。正确写法是描绘**具体的影响力行为**："当她在 Slack 上发出一份分析，至少三个不同团队的 PM 会在当周把这份分析作为自己产品讨论的引用材料"。

**陷阱 4：混淆 Mentor 与 Manager**

IC 专家可以是 mentor——对初级同事做代码评审、回答问题、分享方法论——这是**同伴关系**，不是**汇报关系**。Mentor 没有绩效权、没有晋升决定权、没有分配权。如果画像里出现"对 mentee 做 performance review"，就是把 mentor 写成了 manager。

## 使用本指南的流程

1. 阶段 1 写 01 时，先判断本画像是 TL 画像还是 IC 画像
2. 如果是 IC 画像，写到第五层和第六层之前，**先读本文对应的对照表**
3. 按对照表改写，改写后用自检清单逐条过
4. 写完后通读一遍五-六层，问自己"这位专家如果去做独立咨询，这些职责还成立吗？"——如果大部分成立，说明去管理化做到位了
5. 如果本画像是 TL 画像（明确定位为 Tech Lead 或 Manager），则本文不适用，正常写管理化职责即可
