# 四份文件的模板與撰寫規範

本檔案是 `SKILL.md` 階段 1-3 的執行手冊。它規定了 01/02/03/04 四份文件的層次結構、條目顆粒度、撰寫語氣、篇幅基準與質量紅線。**每生成一份文件之前，先讀本文對應章節，不要憑印象寫。**

---

## 總則：四份文件的共同規範

**一、YAML/元資訊頭部**（每份文件開頭都要有）：

```markdown
# <崗位名> | <文件編號-文件名>

> **適用範圍**：<行業/賽道>，<組織階段：0-1 探索期 / 1-10 規模化早期 / 10-100 擴張期 / 100+ 成熟期>
> **對標水位**：<Staff / Principal / Director>，對應 <Google L6-L7 / Meta IC6-IC7 / 位元組 3-2 等>
> **結構模式**：<嚴格六層 / Path C 主軸+作戰維度>
> **去管理化處理**：<是/否>
> **文件版本**：v1.0 | 生成日期：<YYYY-MM-DD>
> **配套文件**：01-崗位職責 / 02-能力要求 / 03-知識體系 / 04-典型場景
```

**元資訊頭部的關鍵紀律**：組織階段必須明確標註（不是"網際網路公司"而是"網際網路公司 · 100+ 成熟期"），因為同一崗位在不同階段的畫像內容會差出 40-60%。詳見 `組織階段分型.md`。

**二、篇幅基準與逆金字塔條數比例**（參考產品專家、增長專家、資料分析專家已有文件）：

| 文件 | 條目數 | 單條段落 | 總字數 |
|---|---|---|---|
| 01 職責 | **18-28 條** | 3-6 句 + Staff 水位判據 1-2 句 | 4000-7000 字 |
| 02 能力 | **與 01 比例 1 : 1.0~1.3**（約 18-36 條） | 4-8 句 | 5000-9000 字 |
| 03 知識 | **與 01 比例 1 : 1.2~1.5**（約 22-42 條） | 4-10 句 | 6000-11000 字 |
| 04 場景 | 18-22 場景 | 六段式（可選第七段） | 10000-16000 字 |

**逆金字塔原則**：03 知識最厚（一條知識可以相容多種能力）、02 能力次之（一項能力可以相容多種職責）、01 職責最薄（綱要）。**違反此比例的常見錯誤**：能力爆炸型（26:60:80，把職責切碎）、一比一對映型（26:26:26，失去共享底盤）、知識稀薄型（26:30:20，能力沒底層支撐）。超出比例區間時，要麼調粒度，要麼在覆盤裡說明本工種為什麼需要偏離。

**三、統一的條目編號規則**：
- 層/維度 → 大寫阿拉伯數字或羅馬編號（一、二、三 或 I、II、III）
- 條目 → 三級編號（1.1 / 1.2 / 1.3...）
- **禁止跳號**。每份文件生成完成後用 `grep "^### " <file>` 檢查編號連續性。

**四、語氣規範**：
- 使用中文；專業術語首次出現時給出英文原文（例："產品市場契合度（Product-Market Fit, PMF）"）
- 用第三人稱描述"這個專家"或用祈使句描述專家的行為
- **禁止**出現"你應該"、"你需要"這類對 Agent 的直接指令——畫像是"對角色的描繪"，不是"對 Agent 的操作命令"

## 01-崗位職責定義 模板

**回答的核心問題**：這個崗位在組織中"做什麼、對什麼負責"？

### 層次結構

**嚴格六層模式**（單一工種用）：

```
一、戰略層職責（Strategy）
  - 關注"為什麼做"與"往哪裡去"，通常 3-5 條
二、策略層職責（Tactics）
  - 關注"怎麼規劃、怎麼排序、怎麼取捨"，通常 3-5 條
三、執行層職責（Execution）
  - 關注"怎麼把事情做出來、怎麼交付質量"，通常 4-6 條
四、專業深度層職責（Craft Depth）
  - 關注"這個工種的手藝獨到之處"，通常 3-5 條
五、組織層職責（Organization）
  - 關注"怎麼協作、怎麼賦能他人、怎麼建立機制"，通常 3-5 條
六、自進化層職責（Self-Evolution）
  - 關注"怎麼持續成長、怎麼為下一代專家鋪路"，通常 2-4 條
```

**子框架的允許使用**：當某層（特別是執行層）有 5 條以上時，應當用工種領域內的天然分類法做子層次劃分。例如資料分析專家的執行層可以引入 DoorDash 的 3C 子框架（Data Creation / Curation / Consumption），把執行層 6 條職責按"資料創造-資料治理-資料消費"分成三組。**子框架必須來自工種領域內的既有方法論，不得自造**——目的是讓 Agent 在呼叫執行層職責時擁有更細的索引，不是為了顯得結構精緻。詳細示例見 `../../profiles/data-analyst/01-崗位職責定義.md` 第三層。

**去管理化處理**：如果本次畫像選擇了 IC 畫像（去管理化），五層組織層與六層自進化層的內容必須從"權力驅動"改造為"影響力驅動"——例如不寫"管理跨職能團隊"而寫"透過資料洞察影響跨職能的決策走向"。詳細的改寫規則與對照表見 `去管理化畫像指南.md`。

**Path C 複合工種模式**（增長/運營/商業化等跨域崗位）：

```
一、核心主軸職責（Core Spine）
  - 這個複合工種的"作戰母題"，4-6 條
二、作戰維度一：<策略/資料/實驗類>
  - 3-5 條
三、作戰維度二：<市場/獲客/渠道類>
  - 3-5 條
四、作戰維度三：<內容/品牌/敘事類>
  - 3-5 條
五、作戰維度四：<產品驅動/留存/變現類>
  - 3-5 條
六、跨維度職責（Cross-Dimensional）
  - 統籌、文化、組織層面的橫切職責，3-5 條
```

### 單條職責的段落結構

每條職責都要寫成**段落 + Staff 水位判據**，不是"標題+半句話"。推薦結構：

```markdown
### 1.1 <職責名，動賓結構或名詞短語>

<段落開頭一句話點題>。<展開描述：在什麼情境下、面對什麼問題、需要做出哪些判斷與產出>。<為什麼這個職責對本崗位是關鍵的——組織價值或專業價值>。<可選：引用一家頭部公司的具體做法 [E07]，或一個經典方法論作為參照系>。<可選：列出 2-3 個這條職責下的典型任務/交付物>。

**Staff 水位判據**：<1-2 句話寫明 Senior 和 Staff 在這條職責上的核心差別。讓 01 自己就能承擔水位錨定，而不是把水位定義全部推給 02 的"初級 vs 專家對照"。>
```

**關鍵紀律**：每條職責必須帶 Staff 水位判據，否則在終驗收清單裡這條會被標紅。Staff 水位判據是 01 的"自我標尺"——它把"能 vs 不能"的初級線、和"能做高 vs 能做對"的 Staff 線區分開，避免把所有水位差異都推遲到 02。

**示例（產品專家 1.1）**：

> **1.1 定義產品的長期願景與北極星指標**
>
> 產品專家要回答"這個產品五年後應該長成什麼樣、我們用哪一個核心指標來衡量是否走在正確的路上"。這不是寫一份 PPT 願景文件，而是要在市場、使用者、技術三者的交匯處找到"既足夠大、又足夠真、又足夠我們能贏"的敘事，並將其壓縮為一個可追蹤、可拆解、團隊每天都會看的北極星指標（North Star Metric）。Amazon 的"Working Backwards" [E03]、Airbnb 早期的"Belong Anywhere" [E11] 都是這一職責的典範。典型交付物包括：產品願景 narrative、5 年戰略里程碑、北極星指標的定義與拆解樹。
>
> **Staff 水位判據**：Senior 能基於已有方向輸出一份願景文件；Staff 能識別現有願景的隱藏假設、並在多個互競的"看似都對"的方向之間做出有放棄成本的取捨。

### 質量檢查

- [ ] 每條職責都能獨立說清"做什麼、為什麼重要、如何落地"
- [ ] **每條職責都帶 Staff 水位判據小節（1-2 句）**
- [ ] 條目之間沒有重複（每條職責都是獨特的活動範疇）
- [ ] 沒有"良好的 XX 能力"、"具備 XX 眼光"這類空話
- [ ] 至少 30% 的條目引用了業界頭部公司做法或經典框架，且引用統一使用 `[E07]` 格式指向證據庫
- [ ] 條目數 18-28
- [ ] 複合工種或 IC 畫像已按 `去管理化畫像指南.md` 改造五-六層

## 02-能力要求定義 模板

**回答的核心問題**：要完成 01 的職責，需要具備哪些"可遷移的專業能力"？

**關鍵區別**：能力 ≠ 知識。能力是"在情境中做出高質量判斷/產出"的本事；知識是支撐能力的概念、框架、資料。能力通常表達為動詞短語，知識通常表達為名詞概念。

### 層次結構

能力文件通常分為四到五大類：

```
一、思維類能力（Thinking）
  - 判斷、推理、取捨、系統思考等底層思維能力
二、工藝類能力（Craft）
  - 本工種最獨特的、區別於其他工種的"手藝"
三、資料與量化類能力（Data & Quant）
  - 用資料做決策、做實驗、做預測的能力
四、協作與組織類能力（Collaboration & Org）
  - 讓他人也能做出高質量產出的能力
五、元能力（Meta，可選）
  - 學習、反思、自我校準的能力
```

### 單條能力的段落結構

```markdown
### 2.1 <能力名，動詞短語>

<一句話定義這個能力>。<展開描述：這個能力在什麼情境下被呼叫、輸入是什麼、輸出是什麼、什麼樣叫"做得好">。<專家級水位與初級水位的核心差異在哪裡>。<可選：典型的訓練路徑或驗證方法>。

**→ 對應職責**：1.2、3.4
**→ 底層知識**：3.7、3.12
```

**強制紀律——邊寫邊做反向對映**：寫每條能力時，**段末必須當場加上"對應職責"和"底層知識"的反向對映標註**。不允許"先把 02 寫完再回頭補對映"——這樣做的結果幾乎一定是：你會發現某些能力其實沒有真實的職責錨點（孤兒），但因為段落已經寫好，你會傾向於"硬塞一個職責進去"，而這會反向汙染 01。正確做法是：**寫每條能力之前先在心裡點出它對應 01 的哪幾條職責**——如果點不出來，就別寫這條能力。

**示例（增長專家 2.3）**：

> **2.3 設計最小可行實驗以快速證偽假設**
>
> 把一個增長假設拆解為"輸入-處理-輸出"三段式，設計出成本最低、最能快速得出"有效/無效"結論的實驗裝置。專家級水位的標誌是：在寫實驗方案之前，先寫"如果這個假設錯了，我會看到什麼訊號"——即先定義證偽條件，再設計處理組。Reforge 的"Experiment Design"課程與 Sean Ellis 的 High Tempo Testing 框架是這一能力的方法論源頭。底層知識支撐：統計顯著性與檢驗力（見 03 文件 3.12）、實驗設計原則（3.13）、因果推斷（3.14）。初級 vs 專家差異：初級會直接做 A/B 測試並關注結果顯著性，專家會先問"樣本量夠嗎？執行時長夠嗎？有沒有季節性汙染？"
```

### 質量檢查

- [ ] 每項能力都有動詞（"設計"、"判斷"、"權衡"、"拆解"、"溝通"等）
- [ ] **每項能力的段末都有"→ 對應職責"和"→ 底層知識"兩行反向對映**
- [ ] 每項能力明確指向 03 文件中 1-3 個知識條目
- [ ] 每項能力寫出了"專家 vs 初級"的差異點
- [ ] 能力 : 職責的條目數比例落在 **1.0~1.3 : 1**（即 02 略多於 01，但不超過 1.3 倍）

## 03-知識體系定義 模板

**回答的核心問題**：支撐 02 能力的底層"概念、方法論、模型、框架、資料"是什麼？

### 層次結構

知識體系建議按"從抽象到具體"的四層結構：

```
一、基礎理論與思維模型（Foundations）
  - 跨領域的、其他學科借鑑來的、該工種的"世界觀"級概念
  - 例：第一性原理、系統思考、Munger's Latticework、複利效應、反脆弱
二、行業核心框架與方法論（Industry Frameworks）
  - 該工種內部業界公認的經典框架
  - 例（產品）：JTBD、Kano Model、HEART Framework
  - 例（增長）：AARRR、Growth Loops、PLG Flywheel、Four Fits
三、專業工具與度量體系（Tools & Metrics）
  - 該工種日常使用的具體工具、指標、模板
  - 例：北極星指標、留存曲線、佇列分析（Cohort Analysis）
四、行業案例與典範（Case Library）
  - 該工種歷史上的經典案例、反面教材、行業典範
  - 例：Dropbox 推薦系統、Slack 早期增長、Netflix 個性化推薦
```

### 單條知識的段落結構

```markdown
### 3.1 <知識/概念/框架 名稱>（英文原名）

<一段話定義與闡釋>。<這個概念/框架的起源、提出者或代表公司 [E07]>。<它解決什麼問題、在什麼情境下適用>。<常見誤用或反模式>。<延伸閱讀或經典出處>。

**→ 支撐能力**：2.3、2.5
```

**強制紀律——邊寫邊做反向對映**：寫每條知識時，**段末必須當場加上"→ 支撐能力"標註**，至少指向 02 中的 1 條能力。如果一條知識找不到對應的支撐能力，說明它不屬於本工種的"工作記憶"——它可能是相鄰領域的概念，應該刪除而不是硬塞。

**示例（產品專家 3.7）**：

> **3.7 Jobs-To-Be-Done（JTBD，待辦工作理論）**
>
> JTBD 是一種使用者研究與產品定義框架，其核心主張是"使用者並不購買產品，而是僱傭產品來完成某項工作（job）"。由 Clayton Christensen 提出並經 Bob Moesta、Alan Klement 發揚光大。JTBD 的強項是幫助產品團隊跳出"功能/特性"視角，回到使用者的真實動機、情境與情緒層面——尤其適用於市場分析、機會識別、差異化定位三類場景。常見誤用：把 JTBD 等同於"使用者故事"或"用例"；忽略情境與情緒維度只關注功能性 job。經典出處：《Competing Against Luck》（Christensen, 2016）、Bob Moesta 的 Switch 訪談方法。
```

### 質量檢查

- [ ] 四層結構完整，每層條目數 5-12 條
- [ ] 至少 50% 的知識條目引用了具體的人/書/公司/論文，且引用統一使用 `[E07]` 格式指向證據庫
- [ ] **每條知識的段末都有"→ 支撐能力"反向對映**
- [ ] 每條知識都能找到 02 中至少一項能力呼叫它
- [ ] 知識 : 職責的條目數比例落在 **1.2~1.5 : 1**（即 03 是逆金字塔最厚的一層）
- [ ] 沒有寫成"百科詞條"——重點在"在本工種如何使用"，而非通用定義

## 跨文件互參的書寫紀律（強制——違反需要回到對應階段重寫）

三份基礎文件之間必須建立顯式的引用關係，讓 Agent 閱讀時能在"職責-能力-知識"之間自由穿梭。**這不是"建議"也不是"事後補"——是邊寫邊做的硬紀律**。

**強制引用規範**：

| 文件 | 段末必須包含 | 檢查節點 |
|---|---|---|
| 01 職責 | （沒有反向對映，但每條帶 Staff 水位判據） | 寫完一條立即自查"它在 02 是否能找到至少一條能力承載" |
| 02 能力 | `→ 對應職責：1.2、3.4` 與 `→ 底層知識：3.7、3.12` | 寫完一條立即追加對映，否則不進入下一條 |
| 03 知識 | `→ 支撐能力：2.3、2.5` | 寫完一條立即追加對映，找不到對映就刪除該條 |
| 04 場景 | `→ 對應：職責 1.1 / 能力 2.3, 2.8 / 知識 3.7, 3.14`（可選 `相關場景：3.4`） | 寫完一個場景立即追加對映 |

**為什麼必須"邊寫邊對映"而非"先寫完再補"**：

如果等到階段 4 才統一加對映，大機率會出現兩類問題：(a) 部分能力/知識其實沒有真實錨點（孤兒），但因為段落已經寫好，作者會傾向於"硬找一個對應職責塞進去"，反向汙染 01；(b) 大批次補對映會讓作者陷入"對賬模式"而非"創作模式"，注意力消耗在格式整理上而非內容質量上。**所以這條紀律不允許妥協**——一旦發現某個文件沒有邊寫邊對映，必須回到對應階段重寫有問題的條目。

這種顯式引用既是階段 4 交叉驗證的輸入，也是 Agent 在被建立時"回憶自身"的關鍵索引。

## 常見反面示例（要避免的寫法）

**反例 1：空洞套話**
> ❌ "具備優秀的戰略思維和全域性觀，能夠站在更高的維度看問題。"
> ✅ "在多個互競方向間做戰略取捨時，能明確寫出'不做什麼'清單，並說明每項'不做'的機會成本與放棄理由。"

**反例 2：管理化口吻（複合工種）**
> ❌ "作為增長負責人，統籌管理增長團隊的所有工作。"
> ✅ "能獨立操盤從獲客到留存的完整增長引擎，既能自己下場寫廣告文案跑實驗，也能在規模化階段搭建 3-5 人作戰小隊。"

**反例 3：百科詞條化知識**
> ❌ "AARRR 是 Acquisition、Activation、Retention、Revenue、Referral 的縮寫，由 Dave McClure 提出。"
> ✅ "AARRR 是 Dave McClure 針對早期創業公司提出的海盜指標框架。它在種子期-A 輪最有用，幫助團隊建立'漏斗式'增長視角；但在 PMF 之後、進入規模化留存階段時，AARRR 會誤導團隊過度關注頂部漏斗而忽略 Growth Loops 式的複利機制——此時應切換到 Reforge 的 Loops 框架。"

**反例 4：條目過細**
> ❌ 把"寫 PRD"拆成 20 條子職責（"寫標題"、"寫背景"、"寫使用者故事"...）
> ✅ 一條職責涵蓋"完整地把需求翻譯成可執行規格"，子任務放在段落裡描述。

1. 讀本文對應的 01/02/03/04 章節
2. 翻閱 `../../profiles/product-manager/`、`../../profiles/growth-expert/`、`../../profiles/data-analyst/` 的同編號文件作為篇幅和語氣基準

6. 進入下一份文件，寫作時反覆回看前一份以保證對齊
