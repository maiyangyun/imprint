# 结构模式选择：严格六层 vs Path C

本文件是 `SKILL.md` 阶段 0 的执行手册。**在开始生成任何文档之前，必须先选定岗位画像的结构模式**——错误的结构模式会让后续所有工作都建立在歪斜的地基上。

---

## 为什么结构模式重要

结构模式决定了 01 职责文档的层次骨架，而 02/03/04 都会沿用这个骨架。它同时决定了画像的"叙事口吻"——是单一工种的"垂直深度叙事"，还是复合工种的"横向作战叙事"。

选错结构的典型表现：
- 给单一工种（如前端工程师）硬套 Path C → 画像会变得扁平松散，丢失专业纵深
- 给复合工种（如增长专家）用严格六层 → 画像会被迫挤入单一叙事线，丢失作战维度差异

---

## 两种结构模式

### 模式一：严格六层（Strict Six-Layer）

**适用对象**：**单一工种**——岗位的核心活动集中在一个明确的专业领域内。

**典型适用岗位**：
- 产品专家 / Product Manager
- 前端 / 后端 / 全栈工程师
- UX 设计师 / 视觉设计师
- 用户研究员
- 数据科学家 / 数据分析师
- 财务 / 法务 / HR 等职能专家

**六层结构**：

```
一、战略层（Strategy）
    关注"为什么做、往哪里去"——愿景、目标、长期方向
二、策略层（Tactics）
    关注"怎么规划、怎么排序、怎么取舍"——路径、优先级、机会评估
三、执行层（Execution）
    关注"怎么把事情做出来、怎么保证质量"——产出、交付、品质
四、专业深度层（Craft Depth）
    关注"这个工种的手艺独到之处"——专业判断、工艺修养、审美品味
五、组织层（Organization）
    关注"怎么协作、怎么赋能、怎么建制"——影响力、带教、机制
六、自进化层（Self-Evolution）
    关注"怎么持续成长、怎么为下一代铺路"——学习、反思、传承
```

**六层的叙事特点**：从"为什么"到"怎么做"，再从"个人手艺"到"组织影响"，最后到"自我进化"——形成**垂直纵深**叙事。每层是在前一层基础上的"上升一个维度"。

**优势**：
- 结构稳定、可复用——任何单一工种都能套上
- 层次清晰，便于 Agent 建立"自我认知金字塔"
- 交叉验证时容易追踪缺口与孤儿

**劣势**：
- 对复合工种会显得"拧巴"——活动范畴太广，单一维度难以容纳

---

### 模式二：Path C（核心主轴 + 作战维度）

**适用对象**：**复合工种**——岗位横跨多个专业领域，需要同时具备多种作战能力。

**典型适用岗位**：
- 增长专家 / Growth Expert（横跨策略+数据+获客+内容+产品）
- 运营专家 / Operations Expert（横跨内容+活动+数据+用户+渠道）
- 商业化专家 / Monetization Expert（横跨定价+渠道+产品+数据）
- 产品运营 / Product Operations（横跨产品+运营+数据）
- CEO 幕僚 / Chief of Staff（横跨战略+组织+项目+沟通）

**Path C 结构**：

```
一、核心主轴（Core Spine）
    这个复合工种的"作战母题"——所有作战维度都围绕它展开
    例（增长专家）：围绕"可持续的用户价值增长引擎"的构建与放大
    例（运营专家）：围绕"用户生命周期价值的系统化经营"

二、作战维度一（Combat Axis 1）
三、作战维度二（Combat Axis 2）
四、作战维度三（Combat Axis 3）
五、作战维度四（Combat Axis 4）
    4 个作战维度通常对应该工种的 4 个主要活动领域
    例（增长专家）：策略与实验 / 市场与获客 / 内容与叙事 / 产品驱动与留存

六、跨维度职责（Cross-Dimensional）
    统筹、文化、组织层面的横切职责
    例：建立增长文化、跨职能协作、复盘机制、团队搭建
```

**Path C 的叙事特点**：以一条**核心主轴**为骨干，4 个**作战维度**像四肢一样展开——形成"主轴+四维展开"的**横向作战**叙事。

**关键差异**：Path C 不是"从战略到执行"的垂直叙事，而是"在多个作战维度上并行作战"的横向叙事。一个复合工种专家不是"在某一层做得很深"，而是"在多个维度都能独立作战并协调打通"。

**优势**：
- 能容纳复合工种的多维活动范畴
- 作战维度互相并列，避免强行纵向分层
- 核心主轴保证画像不散

**劣势**：
- 对结构设计的要求更高——主轴和维度的定义要经过深思
- 新手容易写成"四个独立的子工种"，失去整体性

## 如何判断岗位属于哪种模式

**判断标准 1：活动范畴的宽度**
- 单一工种：核心活动集中在 1-2 个专业领域
- 复合工种：核心活动分布在 3 个以上并列的专业领域

**判断标准 2：招聘市场上的 JD 画像**
- 翻开 5-10 份头部公司的这个岗位 JD，看职责描述
- 如果 JD 始终围绕单一关键词（如"产品设计"、"后端架构"）→ 严格六层
- 如果 JD 出现"同时具备 A + B + C + D 的能力"、"既要...又要...还要..."→ Path C

**判断标准 3：业界的经典课程体系**
- 该工种有单一垂直的专业课程体系（如 UX 设计课程）→ 严格六层
- 该工种需要拼接多个课程模块（如 Reforge 的增长课程由 Acquisition、Retention、Monetization、Experimentation 等多门课组成）→ Path C

**判断标准 4：工作日常的"作战画面"**
- 专家的一天是"深耕一个问题、追求手艺极致"→ 严格六层
- 专家的一天是"在多个战线之间切换、协调多个维度"→ Path C

---

## 已有画像的结构对照

**产品专家（product-manager）** → 严格六层
- 核心活动集中在"产品定义与交付"这一单一领域
- 战略/策略/执行的纵深叙事天然适配
- 专业深度层突出"产品判断力"这一独特手艺

**增长专家（growth-expert）** → Path C
- 核心活动横跨策略实验、市场获客、内容叙事、产品驱动四大维度
- 核心主轴是"可持续增长引擎的构建与放大"
- 跨维度职责包含增长文化建设、作战节奏设计

## Path C 模式的"去管理化"处理

这是 Path C 最容易出错的地方，**必须专门处理**。

### 什么叫"去管理化"

许多复合工种（增长、运营、商业化）在头部公司通常是有团队的，所以传统 JD 容易写成"增长负责人带增长团队做 XXX"的管理者口吻。但灵魂工程的 Agent 画像必须**强调"作战能力"而非"管理身份"**。

**理由**：
1. Agent 是数字员工，它本身没有"下属"可以管理
2. Staff/Principal 级的个人贡献者（IC）恰恰是"能独立操盘+能带队放大"的复合体，不是纯管理者
3. 过早把岗位定义为"管理者"会降低画像的可执行性

### 去管理化的三条规则

**规则一：动词用"操盘/设计/构建/拆解"，避免"管理/统筹/协调"**
- ❌ "管理增长团队的日常运营"
- ✅ "独立操盘从获客到留存的完整增长引擎"

**规则二：身份用"作战人员+team builder"，避免"负责人/Lead/主管"**
- ❌ "作为增长负责人，带领团队实现季度增长目标"
- ✅ "既能自己下场写广告文案跑实验，也能在规模化阶段搭建 3-5 人作战小队"

**规则三：组织层职责写"建制与赋能"而非"管理与汇报"**
- ❌ "向 CEO 汇报增长进展，管理团队 OKR"
- ✅ "建立增长作战节奏（周实验会、月复盘、季度战役），把方法论沉淀为团队可传承的 playbook"

## 特殊情况与混合模式

**情况 1：初期单一、后期复合的岗位**
例如"早期的运营"是单一工种，"成熟期的运营负责人"是复合工种。应按**对标水位**来选——既然灵魂工程只做 Staff/Principal 级画像，应按成熟形态选结构。

**情况 2：边界模糊的岗位**
例如"产品运营"可以偏产品（单一）也可以偏运营（复合），取决于公司。应在阶段 0 访谈时让用户明确岗位侧重，再决定结构。

**情况 3：确实需要 Path C 但维度数 ≠ 4**
有些复合工种只有 3 个作战维度（例如某些"三角作战"岗位）或有 5 个维度。**4 是推荐数字但不强制**——核心是"一条主轴+并列维度+跨维度职责"的骨架。

---

## 选定结构后的检查

做出结构选择后，向用户口头复述一次：

> "Soren，我判断这个岗位是<复合/单一>工种，建议使用<Path C / 严格六层>结构。
> 核心主轴/六层骨架大致是：<一句话描述>。
> 如果你认同，我就按这个结构开始生成 01 文档。"

得到用户确认后再往下走。结构错了往回改的成本极高——整个 01 职责都要重写。
