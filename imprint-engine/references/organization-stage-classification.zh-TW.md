# 組織階段分型：同一崗位在不同階段的畫像差異

本檔案是 `SKILL.md` 階段 0 的執行手冊（與 `結構模式選擇.md` 並列）。**組織階段是決定畫像內容的核心變數之一**——它決定了同一崗位的職責邊界寬度、基建依賴度、作戰畫面、以及需要呼叫的知識範圍。

**不問清組織階段就開始寫畫像，等於憑感覺拍板。**

---

## 為什麼組織階段重要

在訪談階段，使用者告訴你"要為 XX 崗位做畫像"時，如果沒有問清組織階段，你會預設用"某種平均水位"去寫——通常會偏向成熟公司的分工畫像，因為這是網際網路上 JD 的主流樣本。但這會導致：

- 在初創團隊使用畫像時：Agent 會建議"讓資料團隊搭一下數倉"——可是使用者團隊里根本沒有資料團隊，得自己動手
- 在成熟團隊使用畫像時：Agent 會過度強調"全棧作戰"——可是使用者身處高度分工的組織，反而需要專注在自己的深度

**同一個崗位、同一個水位，在不同階段的畫像可以差出 40-60% 的內容。** 這不是微調，是整個職責重心的遷移。

---

## 四種組織階段的特徵

### 階段 A：0-1 探索期（種子期/天使輪/Pre-A）

**團隊規模**：3-30 人
**典型特徵**：
- 核心命題是"找到 PMF"——產品、市場、商業模式都在驗證中
- 組織邊界模糊，一人身兼多職是常態
- 沒有專職的基建團隊（資料、平臺、DevOps 都是業務團隊自己扛）
- 決策高度依賴創始人直覺+快速迭代，很少有"流程"和"規範"
- 資源極度稀缺，追求"最小投入證偽/證實假設"

**代表公司階段**：A 輪前的早期創業公司、MVP 階段的內部孵化專案

### 階段 B：1-10 規模化早期（A 輪-B 輪）

**團隊規模**：30-150 人
**典型特徵**：
- PMF 初步驗證，開始探索規模化路徑
- 開始出現專業分工，但分工不精細（一個資料分析師可能同時負責 BI、實驗、報表）
- 基建開始搭建但不完善（自建數倉雛形、簡單的實驗工具）
- 流程開始形成但仍然靈活（有 OKR 但不僵化）
- 關鍵動作是"找到規模化的增長路徑+建立可複製的打法"

**代表公司階段**：A 輪-B 輪的成長型公司、剛完成 PMF 的垂直賽道玩家

### 階段 C：10-100 規模化擴張期（C 輪-IPO 前）

**團隊規模**：150-1500 人
**典型特徵**：
- 組織已進入"多業務線並行"階段
- 分工明確、職級體系完善，每個崗位有清晰的職責邊界
- 基建團隊獨立（資料平臺、實驗平臺、CI/CD 團隊成建制）
- 流程開始重要（Code Review、PRD 評審、A/B 規範）
- 關鍵動作是"多業務線協同+組織能力的複製與沉澱"

**代表公司階段**：C 輪-D 輪的擴張期公司、IPO 前的獨角獸

### 階段 D：100+ 成熟期（上市公司/大廠）

**團隊規模**：1500 人以上
**典型特徵**：
- 高度分工，每個崗位只負責一個明確切片
- 基建體系極其完善（資料中臺、實驗平臺、AI 輔助分析工具鏈齊全）
- 流程驅動，決策依賴資料與評審會
- 微小最佳化的價值也值得深度分析（因為流量/使用者基數足夠大）
- 關鍵動作是"在精細化中挖掘增量+在成熟市場中保持創新"

**代表公司階段**：Google、Meta、Amazon、位元組、阿里、騰訊、美團等大廠

## 組織階段對畫像的四類影響

### 影響一：職責邊界寬度

**0-1 階段**：職責邊界模糊，"什麼都要做一點"
**100+ 階段**：職責邊界清晰，"只做自己這一塊但做到極致"

**資料分析專家的對照**：
| 維度 | 0-1 階段 | 100+ 階段 |
|---|---|---|
| 資料採集 | 自己寫埋點、自己做資料治理 | 有資料工程團隊負責 |
| 資料儲存 | 自己搭 BigQuery/ClickHouse 數倉 | 有資料平臺團隊負責 |
| 分析工具 | 自己選型 Metabase/Superset 並維護 | 用公司統一的內部 BI 工具 |
| 實驗 | 自己寫 A/B 分流邏輯 | 用公司實驗平臺，只負責實驗設計 |
| 彙報 | 直接向創始人彙報 | 經過多層彙報鏈 |

### 影響二：基建依賴度

**0-1 階段**：假設"基建不存在"——崗位必須能自己搭建工具鏈
**100+ 階段**：假設"基建是給定的"——崗位只需知道如何使用成熟工具

這決定了 **03 知識文件**的內容範圍：
- 早期階段需要懂"資料埋點規範設計、數倉建模（Kimball 方法論）、BI 工具選型"
- 成熟階段需要懂"實驗平臺的 SRM 診斷、公司內部指標體系、如何與資料團隊協作"

### 影響三：作戰畫面與時間顆粒度

**0-1 階段**：
- 時間顆粒度：以"天"為單位做決策
- 典型一天：早上看昨天的關鍵指標，中午和創始人對一個假設，下午跑一個快速實驗
- 錯了就快速改，重視"學得快"勝過"做得對"

**100+ 階段**：
- 時間顆粒度：以"雙週/月"為單位
- 典型一天：跑 SQL 做深度分析、寫 PRD 級別的分析報告、和 PM/Eng/Design 開評審會
- 錯了代價更高（幾百萬使用者受影響），重視"嚴謹性"勝過"速度"

這決定了 **04 場景文件**的選題和張力：
- 0-1 階段的典型場景："樣本太小，是該下結論還是再等？"
- 100+ 階段的典型場景："實驗顯著但效應量小，是否值得上線？"

### 影響四：組織層與自進化層的內涵

**0-1 階段的組織層**：
- "協作"= 直接和創始人、產品、工程天天對話
- "帶教"=還沒到帶人，重點是自我賦能
- "機制"=不需要重，重點是快速執行

**100+ 階段的組織層**：
- "協作"= 跨部門、跨時區、跨產品線的複雜協同
- "帶教"= 帶初級分析師、建 mentorship 機制
- "機制"= 分析規範、程式碼審查、知識沉澱平臺

## 階段判斷與訪談提問

### 判斷標準

在階段 0 訪談時，用以下訊號確認使用者心中的組織階段：

**訊號 1：使用者描述的工作內容**
- "我要自己搭資料平臺"→ 0-1 或 1-10
- "我們有專門的 X 團隊"→ 10-100 或 100+
- "我向 CEO 直接彙報"→ 0-1 或 1-10
- "我向 Director 彙報，Director 向 VP 彙報"→ 100+

**訊號 2：提到的工具棧**
- "我們用 Metabase/Superset"→ 偏早期
- "我們有內部的 XX 平臺"→ 偏成熟

**訊號 3：描述的決策方式**
- "創始人拍板"→ 0-1
- "OKR 對齊+資料評審"→ 10-100 或 100+

### 訪談提問模板

在階段 0 的多選問題中，把組織階段作為**獨立的必問項**，不要合併到"行業賽道"裡：

```
組織階段（決定同一崗位的職責重心與基建依賴）：
[A] 0-1 探索期（種子/天使，3-30 人，PMF 前）
[B] 1-10 規模化早期（A-B 輪，30-150 人，PMF 已驗證）
[C] 10-100 擴張期（C 輪-IPO 前，150-1500 人，多業務線並行）
[D] 100+ 成熟期（上市公司/大廠，1500+ 人，高度分工）
```

如果使用者選了跨越兩個階段（例如"我們正從 B 輪進入 C 輪"），按**偏向成熟的那一端**寫畫像——因為組織階段的變遷是"能力配備先於組織規模"的。

## 階段對四份文件的具體影響

### 對 01 職責文件

- **戰略層**：0-1 階段更側重"幫助定義公司級別的關鍵假設"；100+ 階段更側重"在既定戰略下最佳化區域性指標"
- **執行層**：0-1 階段條目更多更雜（全棧）；100+ 階段條目更少更深（垂直專精）
- **組織層**：0-1 階段幾乎不涉及"帶人"；100+ 階段是核心職責之一

### 對 02 能力文件

- **工藝類能力**：0-1 階段覆蓋面廣但深度淺；100+ 階段覆蓋面窄但深度極深
- **協作類能力**：0-1 階段強調"直接溝通+快速對齊"；100+ 階段強調"跨組織影響力+向上管理"

### 對 03 知識文件

- **基建類知識**：0-1 階段必須覆蓋基建搭建；100+ 階段只需覆蓋基建使用
- **案例與典範**：引用的公司案例要與目標階段匹配（早期團隊對標 Airbnb 早期的實驗文化，而不是 Google 現在的實驗平臺）

### 對 04 場景文件

- **場景的張力**：0-1 階段張力多來自"資源稀缺+時間壓力"；100+ 階段張力多來自"精度要求+跨組織博弈"
- **場景中的決策空間**：0-1 階段決策空間大（很多未定義變數）；100+ 階段決策空間小（多數引數已鎖定）

## 常見錯誤

**錯誤 1：預設寫成 100+ 階段畫像**
因為網上 JD 和頭部公司資料以大廠為主，不問清階段時 Agent 會自動傾向大廠畫像。這會導致畫像在早期團隊使用時"說的頭頭是道但完全不接地氣"。

**錯誤 2：不敢寫早期階段畫像，怕"不夠專業"**
早期階段的 Staff 級資料分析專家依然是 Staff 級——只是專業性體現為"在極度資源稀缺下依然做出嚴謹判斷"，而不是"用複雜的貝葉斯模型"。不要把"專業"等同於"高大上"。

**錯誤 3：混寫兩個階段**
一份畫像裡前半段按早期寫、後半段按成熟期寫——這是最糟糕的情況，Agent 會產生認知衝突。要選定一個階段，寫完整。

---

## 何時需要生成"跨階段畫像"

罕見情況下，使用者可能希望"同一崗位畫像能跨階段複用"。此時應**拒絕這個需求**，改為建議生成多份階段化畫像（例如 `data-analyst-early` 和 `data-analyst-mature`），以場景文件的維度區分。

強行寫跨階段的通用畫像會讓畫像變得模糊、抽象、失去鋒利度，最終 Agent 在任何階段的團隊都用不好。
