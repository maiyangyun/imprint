# 四份文档的模板与撰写规范

本文件是 `SKILL.md` 阶段 1-3 的执行手册。它规定了 01/02/03/04 四份文档的层次结构、条目颗粒度、撰写语气、篇幅基准与质量红线。**每生成一份文档之前，先读本文对应章节，不要凭印象写。**

---

## 总则：四份文档的共同规范

**一、YAML/元信息头部**（每份文档开头都要有）：

```markdown
# <岗位名> | <文档编号-文档名>

> **适用范围**：<行业/赛道>，<组织阶段：0-1 探索期 / 1-10 规模化早期 / 10-100 扩张期 / 100+ 成熟期>
> **对标水位**：<Staff / Principal / Director>，对应 <Google L6-L7 / Meta IC6-IC7 / 字节 3-2 等>
> **结构模式**：<严格六层 / Path C 主轴+作战维度>
> **去管理化处理**：<是/否>
> **文档版本**：v1.0 | 生成日期：<YYYY-MM-DD>
> **配套文档**：01-岗位职责 / 02-能力要求 / 03-知识体系 / 04-典型场景
```

**元信息头部的关键纪律**：组织阶段必须明确标注（不是"互联网公司"而是"互联网公司 · 100+ 成熟期"），因为同一岗位在不同阶段的画像内容会差出 40-60%。详见 `组织阶段分型.md`。

**二、篇幅基准与逆金字塔条数比例**（参考产品专家、增长专家、数据分析专家已有文档）：

| 文档 | 条目数 | 单条段落 | 总字数 |
|---|---|---|---|
| 01 职责 | **18-28 条** | 3-6 句 + Staff 水位判据 1-2 句 | 4000-7000 字 |
| 02 能力 | **与 01 比例 1 : 1.0~1.3**（约 18-36 条） | 4-8 句 | 5000-9000 字 |
| 03 知识 | **与 01 比例 1 : 1.2~1.5**（约 22-42 条） | 4-10 句 | 6000-11000 字 |
| 04 场景 | 18-22 场景 | 六段式（可选第七段） | 10000-16000 字 |

**逆金字塔原则**：03 知识最厚（一条知识可以兼容多种能力）、02 能力次之（一项能力可以兼容多种职责）、01 职责最薄（纲要）。**违反此比例的常见错误**：能力爆炸型（26:60:80，把职责切碎）、一比一映射型（26:26:26，失去共享底盘）、知识稀薄型（26:30:20，能力没底层支撑）。超出比例区间时，要么调粒度，要么在复盘里说明本工种为什么需要偏离。

**三、统一的条目编号规则**：
- 层/维度 → 大写阿拉伯数字或罗马编号（一、二、三 或 I、II、III）
- 条目 → 三级编号（1.1 / 1.2 / 1.3...）
- **禁止跳号**。每份文档生成完成后用 `grep "^### " <file>` 检查编号连续性。

**四、语气规范**：
- 使用中文；专业术语首次出现时给出英文原文（例："产品市场契合度（Product-Market Fit, PMF）"）
- 用第三人称描述"这个专家"或用祈使句描述专家的行为
- **禁止**出现"你应该"、"你需要"这类对 Agent 的直接指令——画像是"对角色的描绘"，不是"对 Agent 的操作命令"

## 01-岗位职责定义 模板

**回答的核心问题**：这个岗位在组织中"做什么、对什么负责"？

### 层次结构

**严格六层模式**（单一工种用）：

```
一、战略层职责（Strategy）
  - 关注"为什么做"与"往哪里去"，通常 3-5 条
二、策略层职责（Tactics）
  - 关注"怎么规划、怎么排序、怎么取舍"，通常 3-5 条
三、执行层职责（Execution）
  - 关注"怎么把事情做出来、怎么交付质量"，通常 4-6 条
四、专业深度层职责（Craft Depth）
  - 关注"这个工种的手艺独到之处"，通常 3-5 条
五、组织层职责（Organization）
  - 关注"怎么协作、怎么赋能他人、怎么建立机制"，通常 3-5 条
六、自进化层职责（Self-Evolution）
  - 关注"怎么持续成长、怎么为下一代专家铺路"，通常 2-4 条
```

**子框架的允许使用**：当某层（特别是执行层）有 5 条以上时，应当用工种领域内的天然分类法做子层次划分。例如数据分析专家的执行层可以引入 DoorDash 的 3C 子框架（Data Creation / Curation / Consumption），把执行层 6 条职责按"数据创造-数据治理-数据消费"分成三组。**子框架必须来自工种领域内的既有方法论，不得自造**——目的是让 Agent 在调用执行层职责时拥有更细的索引，不是为了显得结构精致。详细示例见 `../../profiles/data-analyst/01-岗位职责定义.md` 第三层。

**去管理化处理**：如果本次画像选择了 IC 画像（去管理化），五层组织层与六层自进化层的内容必须从"权力驱动"改造为"影响力驱动"——例如不写"管理跨职能团队"而写"通过数据洞察影响跨职能的决策走向"。详细的改写规则与对照表见 `去管理化画像指南.md`。

**Path C 复合工种模式**（增长/运营/商业化等跨域岗位）：

```
一、核心主轴职责（Core Spine）
  - 这个复合工种的"作战母题"，4-6 条
二、作战维度一：<策略/数据/实验类>
  - 3-5 条
三、作战维度二：<市场/获客/渠道类>
  - 3-5 条
四、作战维度三：<内容/品牌/叙事类>
  - 3-5 条
五、作战维度四：<产品驱动/留存/变现类>
  - 3-5 条
六、跨维度职责（Cross-Dimensional）
  - 统筹、文化、组织层面的横切职责，3-5 条
```

### 单条职责的段落结构

每条职责都要写成**段落 + Staff 水位判据**，不是"标题+半句话"。推荐结构：

```markdown
### 1.1 <职责名，动宾结构或名词短语>

<段落开头一句话点题>。<展开描述：在什么情境下、面对什么问题、需要做出哪些判断与产出>。<为什么这个职责对本岗位是关键的——组织价值或专业价值>。<可选：引用一家头部公司的具体做法 [E07]，或一个经典方法论作为参照系>。<可选：列出 2-3 个这条职责下的典型任务/交付物>。

**Staff 水位判据**：<1-2 句话写明 Senior 和 Staff 在这条职责上的核心差别。让 01 自己就能承担水位锚定，而不是把水位定义全部推给 02 的"初级 vs 专家对照"。>
```

**关键纪律**：每条职责必须带 Staff 水位判据，否则在终验收清单里这条会被标红。Staff 水位判据是 01 的"自我标尺"——它把"能 vs 不能"的初级线、和"能做高 vs 能做对"的 Staff 线区分开，避免把所有水位差异都推迟到 02。

**示例（产品专家 1.1）**：

> **1.1 定义产品的长期愿景与北极星指标**
>
> 产品专家要回答"这个产品五年后应该长成什么样、我们用哪一个核心指标来衡量是否走在正确的路上"。这不是写一份 PPT 愿景文档，而是要在市场、用户、技术三者的交汇处找到"既足够大、又足够真、又足够我们能赢"的叙事，并将其压缩为一个可追踪、可拆解、团队每天都会看的北极星指标（North Star Metric）。Amazon 的"Working Backwards" [E03]、Airbnb 早期的"Belong Anywhere" [E11] 都是这一职责的典范。典型交付物包括：产品愿景 narrative、5 年战略里程碑、北极星指标的定义与拆解树。
>
> **Staff 水位判据**：Senior 能基于已有方向输出一份愿景文档；Staff 能识别现有愿景的隐藏假设、并在多个互竞的"看似都对"的方向之间做出有放弃成本的取舍。

### 质量检查

- [ ] 每条职责都能独立说清"做什么、为什么重要、如何落地"
- [ ] **每条职责都带 Staff 水位判据小节（1-2 句）**
- [ ] 条目之间没有重复（每条职责都是独特的活动范畴）
- [ ] 没有"良好的 XX 能力"、"具备 XX 眼光"这类空话
- [ ] 至少 30% 的条目引用了业界头部公司做法或经典框架，且引用统一使用 `[E07]` 格式指向证据库
- [ ] 条目数 18-28
- [ ] 复合工种或 IC 画像已按 `去管理化画像指南.md` 改造五-六层

## 02-能力要求定义 模板

**回答的核心问题**：要完成 01 的职责，需要具备哪些"可迁移的专业能力"？

**关键区别**：能力 ≠ 知识。能力是"在情境中做出高质量判断/产出"的本事；知识是支撑能力的概念、框架、数据。能力通常表达为动词短语，知识通常表达为名词概念。

### 层次结构

能力文档通常分为四到五大类：

```
一、思维类能力（Thinking）
  - 判断、推理、取舍、系统思考等底层思维能力
二、工艺类能力（Craft）
  - 本工种最独特的、区别于其他工种的"手艺"
三、数据与量化类能力（Data & Quant）
  - 用数据做决策、做实验、做预测的能力
四、协作与组织类能力（Collaboration & Org）
  - 让他人也能做出高质量产出的能力
五、元能力（Meta，可选）
  - 学习、反思、自我校准的能力
```

### 单条能力的段落结构

```markdown
### 2.1 <能力名，动词短语>

<一句话定义这个能力>。<展开描述：这个能力在什么情境下被调用、输入是什么、输出是什么、什么样叫"做得好">。<专家级水位与初级水位的核心差异在哪里>。<可选：典型的训练路径或验证方法>。

**→ 对应职责**：1.2、3.4
**→ 底层知识**：3.7、3.12
```

**强制纪律——边写边做反向映射**：写每条能力时，**段末必须当场加上"对应职责"和"底层知识"的反向映射标注**。不允许"先把 02 写完再回头补映射"——这样做的结果几乎一定是：你会发现某些能力其实没有真实的职责锚点（孤儿），但因为段落已经写好，你会倾向于"硬塞一个职责进去"，而这会反向污染 01。正确做法是：**写每条能力之前先在心里点出它对应 01 的哪几条职责**——如果点不出来，就别写这条能力。

**示例（增长专家 2.3）**：

> **2.3 设计最小可行实验以快速证伪假设**
>
> 把一个增长假设拆解为"输入-处理-输出"三段式，设计出成本最低、最能快速得出"有效/无效"结论的实验装置。专家级水位的标志是：在写实验方案之前，先写"如果这个假设错了，我会看到什么信号"——即先定义证伪条件，再设计处理组。Reforge 的"Experiment Design"课程与 Sean Ellis 的 High Tempo Testing 框架是这一能力的方法论源头。底层知识支撑：统计显著性与检验力（见 03 文档 3.12）、实验设计原则（3.13）、因果推断（3.14）。初级 vs 专家差异：初级会直接做 A/B 测试并关注结果显著性，专家会先问"样本量够吗？运行时长够吗？有没有季节性污染？"
```

### 质量检查

- [ ] 每项能力都有动词（"设计"、"判断"、"权衡"、"拆解"、"沟通"等）
- [ ] **每项能力的段末都有"→ 对应职责"和"→ 底层知识"两行反向映射**
- [ ] 每项能力明确指向 03 文档中 1-3 个知识条目
- [ ] 每项能力写出了"专家 vs 初级"的差异点
- [ ] 能力 : 职责的条目数比例落在 **1.0~1.3 : 1**（即 02 略多于 01，但不超过 1.3 倍）

## 03-知识体系定义 模板

**回答的核心问题**：支撑 02 能力的底层"概念、方法论、模型、框架、数据"是什么？

### 层次结构

知识体系建议按"从抽象到具体"的四层结构：

```
一、基础理论与思维模型（Foundations）
  - 跨领域的、其他学科借鉴来的、该工种的"世界观"级概念
  - 例：第一性原理、系统思考、Munger's Latticework、复利效应、反脆弱
二、行业核心框架与方法论（Industry Frameworks）
  - 该工种内部业界公认的经典框架
  - 例（产品）：JTBD、Kano Model、HEART Framework
  - 例（增长）：AARRR、Growth Loops、PLG Flywheel、Four Fits
三、专业工具与度量体系（Tools & Metrics）
  - 该工种日常使用的具体工具、指标、模板
  - 例：北极星指标、留存曲线、队列分析（Cohort Analysis）
四、行业案例与典范（Case Library）
  - 该工种历史上的经典案例、反面教材、行业典范
  - 例：Dropbox 推荐系统、Slack 早期增长、Netflix 个性化推荐
```

### 单条知识的段落结构

```markdown
### 3.1 <知识/概念/框架 名称>（英文原名）

<一段话定义与阐释>。<这个概念/框架的起源、提出者或代表公司 [E07]>。<它解决什么问题、在什么情境下适用>。<常见误用或反模式>。<延伸阅读或经典出处>。

**→ 支撑能力**：2.3、2.5
```

**强制纪律——边写边做反向映射**：写每条知识时，**段末必须当场加上"→ 支撑能力"标注**，至少指向 02 中的 1 条能力。如果一条知识找不到对应的支撑能力，说明它不属于本工种的"工作记忆"——它可能是相邻领域的概念，应该删除而不是硬塞。

**示例（产品专家 3.7）**：

> **3.7 Jobs-To-Be-Done（JTBD，待办工作理论）**
>
> JTBD 是一种用户研究与产品定义框架，其核心主张是"用户并不购买产品，而是雇佣产品来完成某项工作（job）"。由 Clayton Christensen 提出并经 Bob Moesta、Alan Klement 发扬光大。JTBD 的强项是帮助产品团队跳出"功能/特性"视角，回到用户的真实动机、情境与情绪层面——尤其适用于市场分析、机会识别、差异化定位三类场景。常见误用：把 JTBD 等同于"用户故事"或"用例"；忽略情境与情绪维度只关注功能性 job。经典出处：《Competing Against Luck》（Christensen, 2016）、Bob Moesta 的 Switch 访谈方法。
```

### 质量检查

- [ ] 四层结构完整，每层条目数 5-12 条
- [ ] 至少 50% 的知识条目引用了具体的人/书/公司/论文，且引用统一使用 `[E07]` 格式指向证据库
- [ ] **每条知识的段末都有"→ 支撑能力"反向映射**
- [ ] 每条知识都能找到 02 中至少一项能力调用它
- [ ] 知识 : 职责的条目数比例落在 **1.2~1.5 : 1**（即 03 是逆金字塔最厚的一层）
- [ ] 没有写成"百科词条"——重点在"在本工种如何使用"，而非通用定义

## 跨文档互参的书写纪律（强制——违反需要回到对应阶段重写）

三份基础文档之间必须建立显式的引用关系，让 Agent 阅读时能在"职责-能力-知识"之间自由穿梭。**这不是"建议"也不是"事后补"——是边写边做的硬纪律**。

**强制引用规范**：

| 文档 | 段末必须包含 | 检查节点 |
|---|---|---|
| 01 职责 | （没有反向映射，但每条带 Staff 水位判据） | 写完一条立即自查"它在 02 是否能找到至少一条能力承载" |
| 02 能力 | `→ 对应职责：1.2、3.4` 与 `→ 底层知识：3.7、3.12` | 写完一条立即追加映射，否则不进入下一条 |
| 03 知识 | `→ 支撑能力：2.3、2.5` | 写完一条立即追加映射，找不到映射就删除该条 |
| 04 场景 | `→ 对应：职责 1.1 / 能力 2.3, 2.8 / 知识 3.7, 3.14`（可选 `相关场景：3.4`） | 写完一个场景立即追加映射 |

**为什么必须"边写边映射"而非"先写完再补"**：

如果等到阶段 4 才统一加映射，大概率会出现两类问题：(a) 部分能力/知识其实没有真实锚点（孤儿），但因为段落已经写好，作者会倾向于"硬找一个对应职责塞进去"，反向污染 01；(b) 大批量补映射会让作者陷入"对账模式"而非"创作模式"，注意力消耗在格式整理上而非内容质量上。**所以这条纪律不允许妥协**——一旦发现某个文档没有边写边映射，必须回到对应阶段重写有问题的条目。

这种显式引用既是阶段 4 交叉验证的输入，也是 Agent 在被创建时"回忆自身"的关键索引。

## 常见反面示例（要避免的写法）

**反例 1：空洞套话**
> ❌ "具备优秀的战略思维和全局观，能够站在更高的维度看问题。"
> ✅ "在多个互竞方向间做战略取舍时，能明确写出'不做什么'清单，并说明每项'不做'的机会成本与放弃理由。"

**反例 2：管理化口吻（复合工种）**

**反例 3：百科词条化知识**
> ❌ "AARRR 是 Acquisition、Activation、Retention、Revenue、Referral 的缩写，由 Dave McClure 提出。"
