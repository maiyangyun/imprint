# 去管理化畫像指南：IC 畫像如何改寫組織層與自進化層

本檔案是 `SKILL.md` 階段 1-3 的執行手冊，專門處理**"去管理化"畫像**（IC Profile，Individual Contributor Profile）的寫作——即畫像明確定位為"一線專家（IC）"而非"經理/主管/Lead"時，**組織層**（第五層）和**自進化層**（第六層）應該怎麼寫。

---

## 為什麼需要這份指南

Agent（包括 Claude）的訓練資料中，"Staff / Principal / Senior" 這類高水位角色的公開資料 60% 以上是**帶管理職責的 TL / Manager 混合角色**——因為英文世界的 "Lead" 一詞本身就在 IC 和 TL 之間搖擺。結果就是：

- 如果不做顯式"去管理化"處理，寫組織層時 Agent 會不自覺地寫出 "帶領團隊"、"管理跨職能專案"、"考核下屬"、"制定團隊 OKR" 這類管理話術
- 寫自進化層時則容易出現 "建立 career ladder"、"參與 performance calibration" 這類管理動作
- 這兩類話術會把一個純粹的 IC 專家畫像汙染成"半管理者半專家"，當 Agent 被部署到沒有團隊的個體使用者環境時，它的行為會預設假設"我有團隊可以呼叫"——這會導致嚴重的角色錯位

本指南的目的就是給出**"管理化 → 去管理化"的對照改寫規則**，讓 IC 畫像的組織層和自進化層也能承擔"影響他人"的作用，但**完全依賴影響力而非權力**。

---

## 核心概念：權力驅動 vs 影響力驅動

| 維度 | 權力驅動（管理化） | 影響力驅動（去管理化） |
|---|---|---|
| **行動起點** | "我決定/我批准/我分配" | "我論證/我示範/我影響" |
| **依賴機制** | 彙報關係、KPI、考核權 | 資料證據、專業聲譽、典範工作、網路關係 |
| **成功標誌** | 團隊產出達標、下屬成長 | 別人主動採納你的方法/觀點，你的工作成為他人參考 |
| **失敗模式** | 管不動、團隊離職 | 沒人引用你、被邊緣化、聲譽消耗 |
| **典型動作** | "我要求所有資料申請統一走新流程" | "我釋出了一份內部白皮書，流程方案裡的資料申請模式被三個團隊主動採納" |

**關鍵原則**：IC 畫像的組織層和自進化層必須全部落在右列。任何一條落在左列的職責/能力，都要麼改寫、要麼刪除。

---

## 改寫對照表：組織層（第五層）

**組織層**在管理化畫像裡通常寫成"團隊建設、跨職能協作、團隊賦能"；在 IC 畫像裡應該改寫為：

| 管理化表達（禁止） | 去管理化改寫（推薦） |
|---|---|
| "帶領 3-5 人資料分析小組完成季度目標" | "透過專家典範工作與方法論文件沉澱，影響 3-5 人規模的同級夥伴採納一致的分析方法論" |
| "制定團隊的技術棧與工具選型決策" | "透過典範案例 + POC 實驗，在同級討論中以證據主導技術棧與工具選型的共識" |
| "對團隊成員進行 1on1 輔導與 performance review" | "對初級同事做 code review / 分析報告同伴評審，留下可回看的反饋記錄，形成事實上的 mentorship" |
| "統籌跨職能專案推進，解決衝突" | "作為跨職能專案的 individual contributor，透過提出更優的共同目標和決策框架來化解衝突，不依賴彙報關係" |
| "建立團隊 OKR 與考核機制" | "把本工種的關鍵度量方法沉澱為可被其他團隊引用的標準，讓'怎麼算好'這件事在組織裡形成共識" |
| "決定團隊招聘與晉升" | "參與同工種的招聘面試與晉升評審，提供一線專家視角的評估依據（無決定權）" |
| "管理團隊預算與資源" | "作為資源申請人，用資料與 ROI 論證為專項工作爭取預算與資源（不直接掌握預算）" |

### 組織層的去管理化典型條目示例

> **5.1 以典範工作影響同級同事**
>
> 作為一線專家，透過持續產出"高密度、可被反覆引用"的工作成果——一份讓團隊爭相引用的資料分析報告、一套讓產品經理願意照做的實驗設計模板、一個讓所有人更新自己做法的儀表盤——讓自己成為團隊預設的"方法論錨點"。這種影響力不靠任命，不靠彙報關係，完全來自工作本身的質量與可複用性。參考 DoorDash 首位資料分析師 Jessica Lachs 在團隊擴張到 50 人之前，如何透過自己寫的實驗方法論文件 [E03] 成為整個資料團隊的事實共識來源。
>
> **Staff 水位判據**：Senior 能寫出高質量的自己用的分析報告；Staff 能把分析報告的方法論抽象成"其他人照著做也能做出同等質量結果"的模板，從而產生規模化影響。

> **5.2 透過資料論證影響跨職能決策**
>
> 面對產品、工程、市場團隊的跨職能決策時，IC 專家的發言權不來自職位高低，而來自"誰的資料證據最紮實、誰的推理鏈條最清晰"。在會議室裡，能用一張圖或一個數字讓所有人停下來重新思考，能指出"我們現在討論的是假設而不是事實"、能提出"讓我們用 X 資料來驗證這一點"。這種以證據為本的發言方式是 IC 專家在組織裡建立影響力的核心通道。
>
> **Staff 水位判據**：Senior 能用資料回答別人問的問題；Staff 能用資料改變別人打算做的決定。

## 改寫對照表：自進化層（第六層）

**自進化層**在管理化畫像裡通常寫成"團隊成長、人才梯隊建設、組織學習機制"；在 IC 畫像裡應該改寫為：

| 管理化表達（禁止） | 去管理化改寫（推薦） |
|---|---|
| "建立團隊的學習機制與知識庫" | "以個人身份持續產出可被他人複用的方法論文件（內部白皮書、模板、case study）" |
| "為團隊成員設計成長路徑" | "對初級同事的工作產出做長期的同伴評審與反饋，讓他人在你身邊更快成長" |
| "推動團隊文化建設" | "透過一致的、可觀察的工作習慣（程式碼評審深度、文件密度、發言方式）在組織裡留下可模仿的專家樣板" |
| "建立 performance calibration 流程" | "在同工種的內部技術分享會里持續輸出前沿話題，成為工種內的資訊聚合點" |
| "培養下一代 Staff/Principal" | "透過 open-source 式的工作輸出（團隊裡的 GitHub repo、社群分享、行業文章）讓後輩可以站在你的肩膀上" |
| "推動團隊技術升級" | "作為早期採納者把新工具/新方法學到位後，寫成 step-by-step 復現指南發到團隊內" |

### 自進化層的去管理化典型條目示例

> **6.1 以個人方法論輸出驅動工種共識**
>
> IC 專家的自進化不靠管理權，而靠"持續產出被他人引用的方法論"。每季度至少產出一份工種內的內部白皮書或方法論覆盤（例如 "本季度我們在實驗設計上踩的三個坑與結構性原因"），讓這些輸出成為工種內部討論時的預設引用源。參考 Airbnb 資料科學團隊 Chetan Sharma 等早期成員的做法 [E11]：不去競爭管理崗，而是讓自己的方法論成為團隊的事實共識。
>
> **Staff 水位判據**：Senior 會分享自己遇到的問題和解法；Staff 會把多個問題抽象成一個結構性框架，讓後來人不必重新踩這些坑。

> **6.2 成為工種內部的資訊聚合點**
>
> 透過持續關注行業前沿（論文、頭部公司的工程部落格、同工種大廠的招聘 JD 變化）並在團隊內做定期分享，成為同事們預設的"問這個問題該找誰"的人。這種"內部學術領袖"身份不靠任命，而靠持續的資訊密度與分享質量。它帶來的附加價值是：當你需要別人協作時，別人已經因為你之前的分享欠了"知識人情"，配合意願天然更高。
>
> **Staff 水位判據**：Senior 會轉發好文章；Staff 會把一組相關文章加工成"這對我們的實際工作意味著什麼"的一頁紙簡報。

## 改寫時的自檢清單

對每一條組織層和自進化層的條目，都應該透過以下自檢：

- [ ] 這條職責是否預設了"我有下屬/我能分配資源"？如是，改寫
- [ ] 這條職責描述的動作是否都是"透過影響力/典範工作/證據"完成的？
- [ ] 如果把這位專家放到一個沒有團隊的個體環境（比如獨立諮詢、創業初期），這條職責還成立嗎？如不成立，改寫
- [ ] 這條職責的成功標誌是否是"別人主動採納你的做法"而不是"團隊指標達成"？
- [ ] 如果換一個專家崗位，這條職責是否可以原樣複製（即它是"IC 影響力"的通用表達）？如是，沒問題；如否，可能摻雜了管理職能

## 常見陷阱

**陷阱 1：隱蔽的管理化表達**

一些動詞雖然沒有直接出現"管理"二字，但暗含權力關係：
- "**驅動**跨職能對齊" → 聽起來是影響力，但 Agent 會傾向於理解為"我有權讓大家對齊" → 改為"透過資料證據和決策框架，在跨職能會議中達成共識"
- "**推動**團隊採納新方法" → "透過 POC 和內部白皮書讓新方法在團隊內形成事實採納"
- "**負責**團隊的技術方向" → "作為團隊內的技術錨點之一，透過深度工作和公開輸出參與形塑團隊技術方向"

**陷阱 2：把 Staff 理解成 TL**

Agent 在訓練資料裡看到的 "Staff Engineer / Staff Analyst" 經常混有 TL 職責。如果 IC 畫像裡出現"Staff 級別應該開始帶人"這類表達，就是被訓練資料汙染了——**純 IC 畫像的 Staff 不帶人**，他/她的影響力來自深度作品，而非彙報關係。

**陷阱 3：把"影響力"寫成空話**

"具備很強的影響力"是空話。正確寫法是描繪**具體的影響力行為**："當她在 Slack 上發出一份分析，至少三個不同團隊的 PM 會在當週把這份分析作為自己產品討論的引用材料"。

**陷阱 4：混淆 Mentor 與 Manager**

IC 專家可以是 mentor——對初級同事做程式碼評審、回答問題、分享方法論——這是**同伴關係**，不是**彙報關係**。Mentor 沒有績效權、沒有晉升決定權、沒有分配權。如果畫像裡出現"對 mentee 做 performance review"，就是把 mentor 寫成了 manager。

## 使用本指南的流程

1. 階段 1 寫 01 時，先判斷本畫像是 TL 畫像還是 IC 畫像
2. 如果是 IC 畫像，寫到第五層和第六層之前，**先讀本文對應的對照表**
3. 按對照表改寫，改寫後用自檢清單逐條過
4. 寫完後通讀一遍五-六層，問自己"這位專家如果去做獨立諮詢，這些職責還成立嗎？"——如果大部分成立，說明去管理化做到位了
5. 如果本畫像是 TL 畫像（明確定位為 Tech Lead 或 Manager），則本文不適用，正常寫管理化職責即可
