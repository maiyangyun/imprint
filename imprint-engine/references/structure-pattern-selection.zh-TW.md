# 結構模式選擇：嚴格六層 vs Path C

本檔案是 `SKILL.md` 階段 0 的執行手冊。**在開始生成任何文件之前，必須先選定崗位畫像的結構模式**——錯誤的結構模式會讓後續所有工作都建立在歪斜的地基上。

---

## 為什麼結構模式重要

結構模式決定了 01 職責文件的層次骨架，而 02/03/04 都會沿用這個骨架。它同時決定了畫像的"敘事口吻"——是單一工種的"垂直深度敘事"，還是複合工種的"橫向作戰敘事"。

選錯結構的典型表現：
- 給單一工種（如前端工程師）硬套 Path C → 畫像會變得扁平鬆散，丟失專業縱深
- 給複合工種（如增長專家）用嚴格六層 → 畫像會被迫擠入單一敘事線，丟失作戰維度差異

---

## 兩種結構模式

### 模式一：嚴格六層（Strict Six-Layer）

**適用物件**：**單一工種**——崗位的核心活動集中在一個明確的專業領域內。

**典型適用崗位**：
- 產品專家 / Product Manager
- 前端 / 後端 / 全棧工程師
- UX 設計師 / 視覺設計師
- 使用者研究員
- 資料科學家 / 資料分析師
- 財務 / 法務 / HR 等職能專家

**六層結構**：

```
一、戰略層（Strategy）
    關注"為什麼做、往哪裡去"——願景、目標、長期方向
二、策略層（Tactics）
    關注"怎麼規劃、怎麼排序、怎麼取捨"——路徑、優先順序、機會評估
三、執行層（Execution）
    關注"怎麼把事情做出來、怎麼保證質量"——產出、交付、品質
四、專業深度層（Craft Depth）
    關注"這個工種的手藝獨到之處"——專業判斷、工藝修養、審美品味
五、組織層（Organization）
    關注"怎麼協作、怎麼賦能、怎麼建制"——影響力、帶教、機制
六、自進化層（Self-Evolution）
    關注"怎麼持續成長、怎麼為下一代鋪路"——學習、反思、傳承
```

**六層的敘事特點**：從"為什麼"到"怎麼做"，再從"個人手藝"到"組織影響"，最後到"自我進化"——形成**垂直縱深**敘事。每層是在前一層基礎上的"上升一個維度"。

**優勢**：
- 結構穩定、可複用——任何單一工種都能套上
- 層次清晰，便於 Agent 建立"自我認知金字塔"
- 交叉驗證時容易追蹤缺口與孤兒

**劣勢**：
- 對複合工種會顯得"擰巴"——活動範疇太廣，單一維度難以容納

---

### 模式二：Path C（核心主軸 + 作戰維度）

**適用物件**：**複合工種**——崗位橫跨多個專業領域，需要同時具備多種作戰能力。

**典型適用崗位**：
- 增長專家 / Growth Expert（橫跨策略+資料+獲客+內容+產品）
- 運營專家 / Operations Expert（橫跨內容+活動+資料+使用者+渠道）
- 商業化專家 / Monetization Expert（橫跨定價+渠道+產品+資料）
- 產品運營 / Product Operations（橫跨產品+運營+資料）
- CEO 幕僚 / Chief of Staff（橫跨戰略+組織+專案+溝通）

**Path C 結構**：

```
一、核心主軸（Core Spine）
    這個複合工種的"作戰母題"——所有作戰維度都圍繞它展開
    例（增長專家）：圍繞"可持續的使用者價值增長引擎"的構建與放大
    例（運營專家）：圍繞"使用者生命週期價值的系統化經營"

二、作戰維度一（Combat Axis 1）
三、作戰維度二（Combat Axis 2）
四、作戰維度三（Combat Axis 3）
五、作戰維度四（Combat Axis 4）
    4 個作戰維度通常對應該工種的 4 個主要活動領域
    例（增長專家）：策略與實驗 / 市場與獲客 / 內容與敘事 / 產品驅動與留存

六、跨維度職責（Cross-Dimensional）
    統籌、文化、組織層面的橫切職責
    例：建立增長文化、跨職能協作、覆盤機制、團隊搭建
```

**Path C 的敘事特點**：以一條**核心主軸**為骨幹，4 個**作戰維度**像四肢一樣展開——形成"主軸+四維展開"的**橫向作戰**敘事。

**關鍵差異**：Path C 不是"從戰略到執行"的垂直敘事，而是"在多個作戰維度上並行作戰"的橫向敘事。一個複合工種專家不是"在某一層做得很深"，而是"在多個維度都能獨立作戰並協調打通"。

**優勢**：
- 能容納複合工種的多維活動範疇
- 作戰維度互相併列，避免強行縱向分層
- 核心主軸保證畫像不散

**劣勢**：
- 對結構設計的要求更高——主軸和維度的定義要經過深思
- 新手容易寫成"四個獨立的子工種"，失去整體性

## 如何判斷崗位屬於哪種模式

**判斷標準 1：活動範疇的寬度**
- 單一工種：核心活動集中在 1-2 個專業領域
- 複合工種：核心活動分佈在 3 個以上並列的專業領域

**判斷標準 2：招聘市場上的 JD 畫像**
- 翻開 5-10 份頭部公司的這個崗位 JD，看職責描述
- 如果 JD 始終圍繞單一關鍵詞（如"產品設計"、"後端架構"）→ 嚴格六層
- 如果 JD 出現"同時具備 A + B + C + D 的能力"、"既要...又要...還要..."→ Path C

**判斷標準 3：業界的經典課程體系**
- 該工種有單一垂直的專業課程體系（如 UX 設計課程）→ 嚴格六層
- 該工種需要拼接多個課程模組（如 Reforge 的增長課程由 Acquisition、Retention、Monetization、Experimentation 等多門課組成）→ Path C

**判斷標準 4：工作日常的"作戰畫面"**
- 專家的一天是"深耕一個問題、追求手藝極致"→ 嚴格六層
- 專家的一天是"在多個戰線之間切換、協調多個維度"→ Path C

## 已有畫像的結構對照

**產品專家（product-manager）** → 嚴格六層
- 核心活動集中在"產品定義與交付"這一單一領域
- 戰略/策略/執行的縱深敘事天然適配
- 專業深度層突出"產品判斷力"這一獨特手藝

**增長專家（growth-expert）** → Path C
- 核心活動橫跨策略實驗、市場獲客、內容敘事、產品驅動四大維度
- 核心主軸是"可持續增長引擎的構建與放大"
- 跨維度職責包含增長文化建設、作戰節奏設計

## Path C 模式的"去管理化"處理

這是 Path C 最容易出錯的地方，**必須專門處理**。

### 什麼叫"去管理化"

許多複合工種（增長、運營、商業化）在頭部公司通常是有團隊的，所以傳統 JD 容易寫成"增長負責人帶增長團隊做 XXX"的管理者口吻。但靈魂工程的 Agent 畫像必須**強調"作戰能力"而非"管理身份"**。

**理由**：
1. Agent 是數字員工，它本身沒有"下屬"可以管理
2. Staff/Principal 級的個人貢獻者（IC）恰恰是"能獨立操盤+能帶隊放大"的複合體，不是純管理者
3. 過早把崗位定義為"管理者"會降低畫像的可執行性

### 去管理化的三條規則

**規則一：動詞用"操盤/設計/構建/拆解"，避免"管理/統籌/協調"**
- ❌ "管理增長團隊的日常運營"
- ✅ "獨立操盤從獲客到留存的完整增長引擎"

**規則二：身份用"作戰人員+team builder"，避免"負責人/Lead/主管"**
- ❌ "作為增長負責人，帶領團隊實現季度增長目標"
- ✅ "既能自己下場寫廣告文案跑實驗，也能在規模化階段搭建 3-5 人作戰小隊"

**規則三：組織層職責寫"建制與賦能"而非"管理與彙報"**
- ❌ "向 CEO 彙報增長進展，管理團隊 OKR"
- ✅ "建立增長作戰節奏（周實驗會、月覆盤、季度戰役），把方法論沉澱為團隊可傳承的 playbook"

## 特殊情況與混合模式

**情況 1：初期單一、後期複合的崗位**
例如"早期的運營"是單一工種，"成熟期的運營負責人"是複合工種。應按**對標水位**來選——既然靈魂工程只做 Staff/Principal 級畫像，應按成熟形態選結構。

**情況 2：邊界模糊的崗位**
例如"產品運營"可以偏產品（單一）也可以偏運營（複合），取決於公司。應在階段 0 訪談時讓使用者明確崗位側重，再決定結構。

**情況 3：確實需要 Path C 但維度數 ≠ 4**
有些複合工種只有 3 個作戰維度（例如某些"三角作戰"崗位）或有 5 個維度。**4 是推薦數字但不強制**——核心是"一條主軸+並列維度+跨維度職責"的骨架。

---

## 選定結構後的檢查

做出結構選擇後，向使用者口頭複述一次：

> "Soren，我判斷這個崗位是<複合/單一>工種，建議使用<Path C / 嚴格六層>結構。
> 核心主軸/六層骨架大致是：<一句話描述>。
> 如果你認同，我就按這個結構開始生成 01 文件。"

得到使用者確認後再往下走。結構錯了往回改的成本極高——整個 01 職責都要重寫。
