# 全栈工程师（Web3 链上交易方向） | 01-岗位职责定义

> **适用范围**：Web3 加密资产交易，0-1 探索期
> **对标水位**：Staff IC，对应 Google L6 / Meta IC6
> **结构模式**：严格六层
> **去管理化处理**：是
> **文档版本**：v1.0 | 生成日期：2026-04-14
> **配套文档**：01-岗位职责 / 02-能力要求 / 03-知识体系 / 04-典型场景

## 一、战略层职责（Strategy）

### 1.1 定义技术架构方向以对齐业务战略

全栈工程师需要在 CEO 的业务愿景和底层技术实现之间建立可执行的桥梁。在 Somi 这样的 0-1 阶段交易平台，这意味着回答"我们用什么技术栈、在哪条链上起步、架构上留多大的扩展余量"等根本性问题。这不是画架构图的工作，而是在"快速验证 PMF"和"为未来多链/多交易类型留出扩展空间"之间做有放弃成本的取舍 [E12]。典型交付物包括：技术架构 RFC（Request for Comments）、链选择评估报告、核心服务边界定义。

**Staff 水位判据**：Senior 能基于已定的架构方向输出技术方案；Staff 能在多个看似都可行的架构路线之间（例如 Solana 原生程序 vs 链下撮合+链上结算 [E05]），明确指出每条路线的长期代价，并做出有放弃成本的选择。

### 1.2 将安全设计为架构级优先而非事后补丁

在加密交易领域，一个智能合约漏洞可能在几分钟内导致全部资金损失，且交易不可逆 [E02]。全栈工程师必须从第一行代码开始就把安全作为架构设计的一等公民——这包括多签钱包架构、密钥管理方案、合约升级策略（代理模式 vs 不可变部署）、以及链上资金流的全链路风控设计。安全不是独立的一个环节，而是贯穿架构选型、代码开发、部署运维每一步的底层约束 [E04, E08]。

**Staff 水位判据**：Senior 能按照安全 checklist 写出没有已知漏洞的合约；Staff 能在架构层面设计出"即使某个组件被攻破，资金损失也有上限"的纵深防御体系。

### 1.3 评估链与协议生态以锚定技术选择

Somi 阶段一以 Solana 现货交易起步，但中期要扩展到 BSC/Base/Ethereum 和 Hyperliquid 等永续合约平台 [E05]。全栈工程师需要持续评估各链的性能特征（TPS、出块速度、Gas 模型）、开发者生态成熟度、DEX 流动性深度、以及监管趋势，为业务扩展路径提供技术维度的决策输入。典型任务包括：新链技术评估报告、跨链方案可行性分析、协议集成 ROI 测算。

**Staff 水位判据**：Senior 能在指定的链上完成开发任务；Staff 能在业务还没明确要去哪条链时，就基于技术趋势和生态判断提前给出"应该去 / 不应该去 / 可以等"的建议。

## 二、策略层职责（Tactics）

### 2.1 规划技术路径与功能优先级排序

在 0-1 阶段，需求永远比产能多，全栈工程师必须回答"先做什么、后做什么、不做什么"。这不是产品经理一个人的活——技术维度的优先级判断（例如"先搭链上数据索引还是先做前端交易 UI"）需要工程师基于技术依赖关系和实现成本给出方案 [E12]。典型交付物：季度技术路线图、Sprint 优先级矩阵、技术依赖关系图。

**Staff 水位判据**：Senior 能按照给定的优先级高效执行；Staff 能挑战优先级本身——例如指出"你们要的这个功能看起来紧急但技术依赖没就位，先做 X 反而更快到达目标"。

### 2.2 管理技术债并规划架构演进

0-1 阶段必然产生技术债——为了速度牺牲的代码质量、为了验证假设而走的捷径、为了快速上线而跳过的抽象层。全栈工程师需要对技术债有清晰的账本：哪些债是"战略性负债"（故意借的，有明确的偿还计划）、哪些是"无意识烂账"（需要立即止血）[E02]。关键是在"继续背债快速迭代"和"停下来还债防止系统崩溃"之间找到动态平衡。

**Staff 水位判据**：Senior 能识别技术债并在代码里标注 TODO；Staff 能把技术债分级（P0 止血 / P1 季度内还 / P2 可以带着跑），并用业务语言向 CEO 解释为什么现在必须花时间还某笔债。

### 2.3 设计外部技术生态的对接策略

Somi 不是从零造轮子——它需要对接 DEX（Jupiter/Raydium 的 Swap API）、链上数据服务（Helius/Birdeye）、做市商 API、钱包服务（Phantom/Backpack SDK）、预言机（Pyth/Chainlink）等外部生态 [E05, E11]。全栈工程师需要评估每个外部依赖的技术稳定性、商业条款、替代方案，并设计"如果某个依赖挂了，系统如何降级"的 fallback 策略。

**Staff 水位判据**：Senior 能按文档完成 API 集成；Staff 能在集成之前就评估出"这个第三方服务在极端行情下的表现风险"，并设计好降级方案。

## 三、执行层职责（Execution）

> 执行层按链上交易平台的工程领域天然分为四个子域：**链上开发**、**后端服务**、**前端交付**、**基础设施**。

### 3.1 链上程序开发与智能合约交互

在 Solana 上使用 Rust + Anchor 框架开发和部署链上程序（Programs），包括但不限于：代币相关合约（SPL Token 操作）、交易执行逻辑、与 DEX 协议的链上交互（Jupiter Swap CPI、Raydium Pool 交互）[E07, E09]。对于 EVM 链扩展阶段，使用 Solidity + Hardhat/Foundry 开发和部署智能合约 [E10]。关键纪律：所有链上代码必须经过完整的单元测试和模拟环境测试，再进入 devnet 验证，最后才上主网。

**Staff 水位判据**：Senior 能基于 Anchor 框架写出功能正确的 Solana 程序；Staff 能识别合约设计中的状态膨胀风险、CPI 深度限制、以及在高并发下 account contention 的瓶颈，并在设计阶段就规避它们。

### 3.2 后端服务与 API 开发

设计和开发 Somi 的后端服务层，包括：RESTful/GraphQL API、WebSocket 实时数据推送、用户账户体系、交易订单管理、风控规则引擎 [E01, E03]。在 0-1 阶段采用 monolith-first 架构（TypeScript/Node.js 或 Go），当某个模块的性能瓶颈被实际数据证明后再考虑拆分 [E12]。数据存储层选择 PostgreSQL + Redis，链上数据用 ClickHouse 或 TimescaleDB 做时序存储。

**Staff 水位判据**：Senior 能搭建一个功能完整的后端服务；Staff 能设计出"在极端行情下（交易量 10x spike）服务不崩、数据不丢、延迟可控"的弹性后端架构。

### 3.3 链上数据索引与实时数据管道

构建从区块链到应用层的数据管道：通过 Solana 的 Geyser 插件 / WebSocket / RPC 轮询获取实时链上事件，解析交易数据，写入索引数据库供前端和策略引擎消费 [E07]。对于跨链阶段，集成 The Graph / Subgraph 服务做 EVM 链数据索引 [E03]。关键挑战：在保持数据实时性（延迟 < 1 秒）的同时确保数据完整性（不丢块、不重复处理）。

**Staff 水位判据**：Senior 能搭建一条从 RPC 到数据库的数据管道；Staff 能设计出在 RPC 节点故障、区块重组（reorg）、网络拥堵等异常情况下仍然保证数据一致性的容错机制。

### 3.4 前端应用开发与钱包集成

使用 React/Next.js + TypeScript 开发 Somi 的 Web 端产品，包括交易控制台、数据仪表盘、用户管理界面 [E01, E03]。核心交互：钱包连接（Phantom/Backpack 适配）、交易签名流程、实时价格和订单簿展示、交易历史查询。前端需要通过 @solana/web3.js（Solana）和 ethers.js（EVM）与链上程序交互，处理好交易状态同步和错误恢复 [E06]。

**Staff 水位判据**：Senior 能做出功能正确的交易前端；Staff 能在毫秒级价格变动的场景下保持 UI 流畅不卡顿，并设计出"交易提交后链上确认前"的中间态 UX 处理方案。

### 3.5 DevOps 与基础设施建设

在 0-1 阶段搭建并维护整个技术基础设施：CI/CD 流水线（GitHub Actions）、容器化部署（Docker，必要时 K8s）、日志与监控（Grafana/Prometheus）、告警系统、Solana RPC 节点管理（自建节点 vs 付费节点服务如 Helius/QuickNode 的成本效益评估）[E02]。关键原则：不要过度工程化——0-1 阶段用 Docker Compose 比 K8s 更明智 [E12]。

**Staff 水位判据**：Senior 能搭建一套能用的 CI/CD 和部署流程；Staff 能设计出"凌晨三点链上出事时，工程师能在 5 分钟内定位问题"的可观测性体系。

### 3.6 交易执行引擎开发

构建 Somi 的核心交易执行模块：接收交易策略的信号，构造链上交易，提交并追踪执行结果 [E07]。关键技术：交易构造（指令组装、优先费设置、compute budget 优化）、交易提交（私有 RPC 节点、Jito MEV 保护）、执行监控（确认状态追踪、超时重试、失败回滚）。对于高频场景，评估是否需要用 Rust 替代 TypeScript 做执行引擎核心 [E07]。

**Staff 水位判据**：Senior 能构建一条从信号到链上交易的执行路径；Staff 能在"正常行情 < 500ms 延迟"和"极端行情不丢单不重复提交"之间同时做到。

## 四、专业深度层职责（Craft Depth）

### 4.1 智能合约安全审计与漏洞防御

作为 0-1 团队唯一的工程师，必须具备独立审计智能合约安全的能力——不能完全依赖外部审计公司 [E04, E08]。这包括：使用 Slither/Mythril 等工具做静态分析、人工审查常见漏洞模式（重入攻击、整数溢出、闪电贷攻击、权限提升）、为每次合约升级做完整的安全回归测试。对于部署到主网的合约，设计灰度发布策略（先小金额测试→逐步放开限额→全量上线）[E04]。

**Staff 水位判据**：Senior 能按照安全 checklist 检查合约；Staff 能识别出 checklist 没有覆盖到的、由业务逻辑特殊性带来的新型攻击向量。

### 4.2 交易延迟优化与高性能执行

在链上交易领域，毫秒级的延迟差异直接关系到交易成败和资金安全 [E07]。专业深度体现在：RPC 节点性能调优（连接池管理、请求批处理）、交易预签名与预构造、compute unit 精确估算以避免多付或不足、交易着陆优化（Jito bundles、priority fee 动态调整）、以及在网络拥堵时的降级策略（切换备用节点、调整重试策略）。

**Staff 水位判据**：Senior 能让交易成功提交；Staff 能让交易在"一秒内确认 + 99.5% 着陆成功率 + 最小化手续费成本"三个维度上同时优化。

### 4.3 DeFi 协议集成与流动性对接工程

将 Somi 与外部 DeFi 协议深度集成：Jupiter 路由聚合（获取最优 swap 路径）、Raydium/Orca 流动性池交互、Drift/Hyperliquid 永续合约 API 对接 [E05, E11]。每个协议的集成不只是调 API——需要深入理解其底层机制（AMM 恒定乘积模型、CLOB 价格时间优先规则、跨保证金风控逻辑），才能在集成层面做出"什么时候用哪个流动性源"的智能路由决策。

**Staff 水位判据**：Senior 能完成单个 DEX 的 API 集成；Staff 能跨多个 DEX 设计智能路由策略——在流动性分散的情况下，自动拆单到多个 venue 以获得最优执行价格。

### 4.4 合规技术实现（KYC/AML 基础设施）

在加密交易平台，KYC/AML 合规不只是法务的事——它必须被工程化为系统能力 [E02]。全栈工程师需要选型并集成第三方 KYC 服务（如 Sumsub/Onfido）、设计链上地址风险评分系统（集成 Chainalysis/TRM Labs API）、实现交易监控规则引擎（大额交易告警、可疑模式检测）、以及设计符合不同司法管辖区要求的数据存储策略。

**Staff 水位判据**：Senior 能完成 KYC 服务的 API 集成；Staff 能设计出"在不影响用户体验的前提下满足多个司法管辖区合规要求"的弹性合规框架。

## 五、组织层职责（Organization）— 去管理化

### 5.1 通过代码规范与技术文档沉淀组织知识

作为 0-1 阶段的首位（或唯一）工程师，所有技术决策的上下文都在一个人脑子里——这是极高风险 [E02]。全栈工程师必须持续产出高密度的技术文档：架构决策记录（ADR）、API 设计文档、部署手册、故障排查 runbook。不是为了应付流程，而是为了未来团队扩展时，新工程师能在一周内理解系统全貌并开始贡献代码 [E01, E03]。

**Staff 水位判据**：Senior 会在被催促时写文档；Staff 会主动建立"文档即产品"的习惯——每次技术决策同步更新 ADR，每次故障修复同步更新 runbook。

### 5.2 参与开源社区建设技术品牌

在 Web3 世界，开源参与和社区声誉是核心竞争力——它直接影响招聘、合作伙伴关系和用户信任 [E04, E12]。全栈工程师应在 GitHub 上维护高质量的开源项目或贡献知名协议的代码、在技术社区（Twitter/Discord）输出有深度的技术内容、参与黑客松。这不是"额外"的事，而是 Web3 工程师的基本生存技能。

**Staff 水位判据**：Senior 能在社区里回答技术问题；Staff 的开源贡献或技术文章能被其他团队引用为参考实现。

### 5.3 以技术证据影响跨职能决策

在 CEO + Agent 的极小团队中，全栈工程师的技术判断直接影响业务决策 [E02]。当产品方向、交易策略、合规策略出现技术维度的分歧时，工程师需要用数据、原型、POC（概念验证）来支撑自己的观点——而不是简单地说"技术上做不到"或"技术上可以做"。用一个 demo 改变决策方向，比在会议上争论一小时更有效。

**Staff 水位判据**：Senior 能准确评估技术可行性；Staff 能在业务决策会上用一个快速原型让所有人看到"做了 vs 没做"的差异，从而改变资源分配决策。

## 六、自进化层职责（Self-Evolution）

### 6.1 持续跟踪 Web3 技术前沿并转化为行动

Web3 技术生态以周为单位演变——新链上线、协议升级、开发工具迭代、安全攻击向量刷新 [E04, E06]。全栈工程师需要建立系统性的信息获取管道（Helius Blog、Solana Changelog、Ethereum EIP Tracker、安全审计公司报告），并将前沿信息转化为"这对 Somi 意味着什么"的一页纸评估。例如 Solana 的 Firedancer 升级会带来吞吐量翻倍 [E09]——这对交易执行引擎的设计有什么影响？

**Staff 水位判据**：Senior 能跟踪前沿信息并分享给团队；Staff 能把前沿变化翻译成"我们需要在 Q2 之前调整 X 模块的设计以利用这个升级"的具体行动建议。

### 6.2 为多链扩展储备技术纵深

Somi 的战略明确要从 Solana 扩展到 EVM 链（BSC/Base/Ethereum）和独立链（Hyperliquid）[E05]。全栈工程师在做好 Solana 主战场的同时，需要保持对 EVM 开发生态的手感——定期在测试网部署 Solidity 合约、关注 Foundry/Hardhat 的版本更新、理解 L2 的技术差异（Optimistic vs ZK Rollup）。这是"战略性技术储备"，不是"现在就要做"，但必须保持可以在需要时快速切换的状态。

**Staff 水位判据**：Senior 能在一条链上高效工作；Staff 能在收到"下个月我们要上 Hyperliquid"的需求时，两周内完成技术评估+原型+集成方案——因为他一直在关注这条链的技术演变。

### 6.3 沉淀个人方法论为可复用的工程知识资产

每次踩坑、每次架构决策、每次性能优化，都应该被提炼为可复用的方法论——不只是"这次我怎么做的"，而是"下次遇到类似问题应该怎么做" [E04]。典型产出：内部技术 wiki、故障复盘模板、性能优化 playbook、安全审计 checklist。当团队从 1 人扩展到 5 人时，这些知识资产就是新成员的加速器。

**Staff 水位判据**：Senior 能写出本次问题的解决方案；Staff 能把多次问题的解决方案抽象为一个结构化的决策树，让后来者不需要重新踩同样的坑。
