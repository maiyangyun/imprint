## 一、基础理论与思维模型（Foundations）

### 3.1 第一性原理与系统思考

从最基本的事实和约束出发推导解决方案，而非类比已有经验。在 Web3 工程中，这意味着不被"别人都这么做"束缚——例如当所有人都用 AMM 做交易时，Hyperliquid 从第一性原理出发选择了自建 L1 + 链上 CLOB [E05]。系统思考则要求工程师理解改动一个组件时对整个系统的连锁反应——修改合约的状态结构，会影响前端的数据展示、后端的索引逻辑、以及监控告警的阈值设定。

**→ 支撑能力**：2.1、2.4、2.11、2.15

### 3.2 密码学基础

不需要能证明椭圆曲线的安全性，但必须理解：公私钥体系（Ed25519 / secp256k1）、数字签名机制、哈希函数特性（SHA-256 / Keccak-256）、默克尔树与默克尔证明、零知识证明的基本概念（ZK-SNARK / ZK-STARK）。这些是理解区块链共识机制和智能合约安全的底层语言 [E06]。

**→ 支撑能力**：2.2、2.5、2.6

### 3.3 软件架构设计原则

SOLID 原则、领域驱动设计（DDD）基本概念、微服务 vs 单体架构的取舍、事件驱动架构（EDA）、CQRS 模式。在 0-1 阶段，最重要的不是精通所有架构模式，而是理解"monolith-first + 按需拆分"的哲学 [E12]——知道什么时候单体足够好、什么信号表明必须开始拆分。

**→ 支撑能力**：2.1、2.3、2.7

### 3.4 工程效能与 DevOps 基础

CI/CD 流水线设计（GitHub Actions）、容器化（Docker/Docker Compose）、基础设施即代码（Terraform 基础）、日志与监控（Prometheus/Grafana/Loki）、告警设计原则。以及更根本的：如何写出可测试的代码、如何设计有意义的集成测试、如何建立"每次提交都自动验证"的工程习惯 [E02]。

**→ 支撑能力**：2.3、2.10、2.12、2.15

## 二、区块链核心知识（Blockchain Core）

### 3.5 Solana 架构与运行机制

Solana 的核心创新：Proof of History（PoH）时钟机制、Tower BFT 共识、Gulf Stream 交易转发、Turbine 块传播、Sealevel 并行执行引擎。账户模型（与 EVM 状态模型的本质差异）：Solana 程序是无状态的、数据存储在独立的账户中、程序通过 CPI 互相调用。交易生命周期：从客户端构造→RPC 转发→Leader 打包→确认→最终性。理解 Firedancer 升级的影响：独立的验证者客户端实现，吞吐量和可靠性双重提升 [E09]。

**→ 支撑能力**：2.5、2.9

### 3.6 Anchor 框架深度

Anchor 不只是一个简化开发的工具——理解它的设计哲学：(1) 账户验证宏（#[account]）如何消除手写反序列化；(2) 指令上下文（Context<T>）如何强制类型安全；(3) IDL 自动生成如何桥接 Rust 和 TypeScript；(4) Error 处理与自定义错误码；(5) PDA（Program Derived Address）设计模式——何时用 PDA、seeds 如何选择、PDA 的确定性与安全性 [E09]。以及 Anchor 的局限：性能敏感场景可能需要绕过 Anchor 用原生 Solana SDK。

**→ 支撑能力**：2.5、2.9

### 3.7 Solana 数据获取生态

三种获取链上数据的方式及其取舍：(1) JSON-RPC 轮询——最简单但延迟高、负载大；(2) WebSocket 订阅——实时但连接不稳定、miss 风险；(3) Geyser 插件——最低延迟、最稳定、但需要自建或付费节点 [E07]。以及上层聚合服务：Helius DAS API、Birdeye Price API、Jupiter Quote API。每种方式的延迟、成本、可靠性特征，以及在不同业务场景（实时交易 vs 历史分析 vs 用户展示）下的选择。

**→ 支撑能力**：2.8、2.5

### 3.8 EVM 与 Solidity 深度

Ethereum Virtual Machine 的执行模型：栈式虚拟机、Gas 计量机制、存储布局（slot mapping）、内存模型。Solidity 进阶：继承与接口设计、Library 的 delegatecall 机制、事件与日志、ABI 编码规则。Gas 优化技术：storage vs memory vs calldata 选择、紧凑存储打包、短路求值。代理合约模式（UUPS / Transparent Proxy / Diamond）及其安全隐患 [E10]。

**→ 支撑能力**：2.6

### 3.9 智能合约安全知识体系

已知漏洞模式全景：重入攻击（包括跨函数重入）、整数溢出/下溢（Solidity 0.8+ 内置检查 vs 低版本风险）、闪电贷攻击（价格操纵）、委托调用风险、访问控制缺失、前后运行攻击（front-running / sandwich）[E08]。安全开发实践：OpenZeppelin 合约库使用规范、Slither/Mythril 静态分析工具、Foundry fuzz testing、形式化验证（Act/Certora）基础。安全审计流程：内部审查→自动化工具→外部审计→部署安全（多签+时间锁+灰度）。

**→ 支撑能力**：2.2、2.6

### 3.10 Solana 程序安全专项

与 EVM 安全不同的 Solana 特有攻击面：(1) 账户所有权检查缺失——程序必须验证传入账户确实由预期程序拥有；(2) Signer 检查缺失——关键操作必须验证签名者身份；(3) PDA 碰撞——seeds 选择不当可能导致不同用户的 PDA 冲突；(4) CPI 深度炸弹——恶意合约通过深层 CPI 调用耗尽 compute budget；(5) 时间依赖——使用 Clock sysvar 而非 slot 作为时间源的风险 [E08, E09]。

**→ 支撑能力**：2.2、2.5

## 三、DeFi 交易专业知识（DeFi & Trading）

### 3.11 AMM 机制深度

自动做市商的数学模型与工程实现：恒定乘积模型（Uniswap V2: x·y=k）、集中流动性（Uniswap V3: 自定义价格区间）、稳定币专用曲线（Curve StableSwap: 恒定和+恒定积混合）。Solana 生态的 AMM 实现：Raydium（基于 OpenBook 的混合 AMM）、Orca（Whirlpool 集中流动性）、Meteora（动态费用）。理解无常损失（Impermanent Loss）的机制和对流动性提供者的影响 [E11]。

**→ 支撑能力**：2.4、2.9

### 3.12 链上订单簿与混合流动性模型

链上 CLOB（Central Limit Order Book）的技术实现：OpenBook/Serum 的链上订单簿设计、Hyperliquid 的 HyperCore 原生交易引擎（价格-时间优先规则、20 万 TPS）[E05]。混合流动性模型：Drift Protocol 的三层设计——JIT 拍卖（做市商实时报价）+ DLOB（去中心化限价订单簿）+ AMM（兜底流动性）[E05]。理解不同模型的延迟特征、流动性深度特征和适用场景。

**→ 支撑能力**：2.4、2.9

### 3.13 DEX 聚合与智能路由

Jupiter 的路由优化原理：扫描多个流动性源（Raydium/Orca/Meteora/Lifinity）、Metis 引擎动态刷新路径、拆单策略（大单拆到多个 venue 降低价格冲击）[E05]。MEV（Maximal Extractable Value）与三明治攻击：攻击原理、检测方法、防护策略（Jupiter ShadowLane / 私有 RPC / Jito bundles）。预言机集成：Pyth Network（拉取式价格源）vs Chainlink（推送式价格源）的架构差异和选择逻辑 [E11]。

**→ 支撑能力**：2.4、2.9

### 3.14 永续合约与衍生品交易基础

Somi 阶段二将扩展到链上永续合约交易，需要理解：永续合约的资金费率（Funding Rate）机制、保证金体系（隔离保证金 vs 跨保证金 [E05]）、清算机制、标记价格 vs 最后成交价的设计。技术层面：Hyperliquid 的双层架构如何实现高性能衍生品交易、Drift 的跨保证金风控引擎设计 [E05]。这是"现在不做但必须懂"的储备知识。

**→ 支撑能力**：2.4

## 四、工程工具链知识（Tooling & Infrastructure）

### 3.15 交易执行技术栈

从策略信号到链上交易的全链路技术：(1) @solana/web3.js 交易构造——指令组装、签名、序列化；(2) 优先费（priority fee）动态计算——基于最近区块的费用分布；(3) Compute Budget 程序——请求额外计算单元；(4) Jito MEV 保护——通过 bundle 提交避免被三明治攻击；(5) 交易着陆监控——confirmTransaction vs getSignatureStatuses 的使用场景；(6) 重试策略——指数退避+最大重试次数+交易过期检查（blockhash 有效期）[E07]。

**→ 支撑能力**：2.9

### 3.16 前端 Web3 集成知识

钱包适配层：Solana Wallet Adapter（统一接入 Phantom/Backpack/Solflare 等）、ethers.js/wagmi（EVM 链钱包连接）。交易签名 UX：如何设计"签名前预览交易内容"、"签名后追踪确认状态"、"失败后友好提示"的用户体验。实时数据展示：WebSocket 价格推送、K 线渲染（lightweight-charts）、订单簿深度图。响应式设计在交易场景下的特殊挑战：数据密集型界面的移动端适配 [E01, E06]。

**→ 支撑能力**：2.7

### 3.17 数据存储与索引技术

关系型数据库（PostgreSQL）在交易平台的应用：订单表设计、交易历史索引、用户资产快照。时序数据库（ClickHouse/TimescaleDB）在链上数据分析的应用：交易量聚合、价格 OHLCV 计算、链上活动趋势。缓存层（Redis）：实时价格缓存、用户会话管理、速率限制。链上数据索引服务：The Graph 的 Subgraph 机制（EVM 链）、自建索引服务 vs 付费服务（Helius DAS API）的成本效益比较 [E03, E07]。

**→ 支撑能力**：2.8、2.7

## 五、Web3 生态与行业知识（Ecosystem）

### 3.18 Solana 生态全景

核心基础设施：验证者网络、RPC 节点服务商（Helius/QuickNode/Triton）、钱包生态（Phantom/Backpack）。DeFi 协议：Jupiter（聚合器）、Raydium/Orca（AMM）、Drift（永续）、Marinade（质押）。开发者工具：Anchor/Seahorse/Poseidon、Solana CLI、Explorer（Solscan/SolanaFM）。新兴方向：Firedancer（验证者客户端）、压缩 NFT、Token Extensions（Token-2022）。了解这个生态图谱的意义在于：知道"不需要自己造什么"和"生态里缺什么可能是机会" [E09]。

**→ 支撑能力**：2.5、2.14

### 3.19 EVM 生态与 L2 格局

Ethereum 主网、L2（Arbitrum/Optimism/Base/zkSync）的技术差异：Optimistic Rollup vs ZK Rollup 的安全模型和确认延迟差异。EVM 兼容链（BSC/Polygon）的定位和生态特征。开发者工具链对比：Hardhat（JS 生态）vs Foundry（Rust 实现、原生 Solidity 测试）[E10]。Somi 阶段二扩展时需要评估"先上哪条 EVM 链"的关键维度：用户基数、DEX 流动性、Gas 成本、开发者体验 [E06]。

**→ 支撑能力**：2.6、2.14

### 3.20 代币经济学（Tokenomics）基础

不需要设计代币模型，但必须理解：代币标准（ERC-20/SPL Token）、代币发行与分发机制、质押经济学、治理代币、代币激励设计的常见陷阱。工程师理解 tokenomics 的意义在于：当 CEO 说"我们要发一个代币"时，你能评估这对技术架构意味着什么——合约设计、发行流程、链上治理、以及对现有系统的侵入性 [E12]。

**→ 支撑能力**：2.1、2.11

### 3.21 加密合规技术知识

KYC/AML 技术实现方案：第三方 KYC 服务（Sumsub/Onfido）的集成模式、链上地址风险评分（Chainalysis/TRM Labs）、交易监控规则引擎设计。不同司法管辖区（香港/美国/EU）的合规差异及其对技术架构的影响。隐私合规（GDPR）对数据存储和用户数据处理的约束 [E02]。

**→ 支撑能力**：2.1

### 3.22 Web3 技术信息获取渠道

核心信息源：Solana Changelog（协议更新）、Ethereum EIP Tracker（提案追踪）、Helius Blog（Solana 技术深度）、Rekt News（安全事件）、Paradigm Research（前沿研究）。社区渠道：Twitter/X Crypto Tech 圈、Discord 开发者频道、Telegram 技术群。学习资源：Cyfrin Updraft（Solidity 安全）、Solana Cookbook（开发参考）、Anchor Book。建立每周 2-3 小时的固定信息扫描节奏 [E04, E06]。

**→ 支撑能力**：2.13、2.14
