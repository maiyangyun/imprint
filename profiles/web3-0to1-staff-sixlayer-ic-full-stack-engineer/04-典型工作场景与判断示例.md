## 一、战略层场景（3 个）

### 场景 1.1 交易架构选型：链上撮合 vs 链下撮合+链上结算

**场景描述**：Somi MVP 启动第 2 周。CEO 要求 6 周内上线现货交易功能。方案 A：完全链上，对接 Jupiter/Raydium swap；方案 B：链下撮合+链上结算，自建订单簿。CEO 倾向 B，"这样我们能控制体验"。你手上有 Drift 和 Jupiter 的架构文档。

**关键信号**：6 周 + 1 人团队 → 复杂度受控；CEO 倾向 B 但自建撮合需 10+ 周；核心目标是 PMF 验证；Jupiter 已有路由优化+MEV 保护 [E05]。

**决策选项**：A. 链下撮合+链上结算（10-12 周）*捍卫者*：传统金融背景工程师。B. 完全链上集成 Jupiter（3-4 周）*捍卫者*：Web3 原生思维。C. 混合方案（6-8 周）*捍卫者*：风险对冲思维。

**判断逻辑**：专家选 B。0-1 阶段首要目标是 PMF 验证（职责 1.1、知识 3.3）；Jupiter 路由优化+MEV 保护已超过自建质量（知识 3.13）；1 人 6 周无法同时搭撮合引擎和 UI（能力 2.1）；B 保留最大灵活性（职责 2.2）。对 CEO："用 Jupiter 4 周上线，先验证用户是否愿意交易，再决定是否自建"（能力 2.11）。

**常见陷阱**：过度工程化；低估集成复杂度；妥协走 C 导致两边都做不好。

**新手 vs 专家**：新手开始评估撮合方案 → 专家先问"这个阶段目标是什么"；新手试图一次做到位 → 专家做最小方案留扩展点。

**→ 对应**：职责 1.1, 2.1 / 能力 2.1, 2.11 / 知识 3.3, 3.13 | **→ 相关场景**：2.1, 4.3

### 场景 1.2 安全架构：合约部署前的安全保障级别

**场景描述**：Somi 首个涉及资金的 vault 合约开发完毕，50+ 单元测试通过。CEO 希望下周一上主网。外部审计报价 $30K、需 3-4 周。你需要决定无外审上线是否可接受。

**关键信号**：涉及用户资金且不可逆 [E02]；$30K 对初创公司不小但安全事故损失更大；合约功能简单（存/取）攻击面有限；无其他工程师 code review。

**决策选项**：A. 等外审完成（+4 周 $30K）。B. 设金额上限后立即上线，同步做外审。C. 用 Slither + 人工复查替代外审。

**判断逻辑**：专家选 B。设存入上限（单用户 $500，总 TVL $10K）控制风险敞口（能力 2.2）；同步启动外审，通过后逐步提限——灰度发布 [E04]（知识 3.9）；部署用 multisig + pause 功能；Slither + fuzz test 作补充不替代外审。

**常见陷阱**：不设限额直接上线；被 $30K 劝退跳过审计；忽视 pause 机制。

**新手 vs 专家**：新手"测试全过了" → 专家"测试覆盖已知但不覆盖未知"；新手二选一上或不上 → 专家用限额+multisig+pause 建纵深防线。

**→ 对应**：职责 1.2, 4.1 / 能力 2.2 / 知识 3.9, 3.10 | **→ 相关场景**：4.1

### 场景 1.3 新链评估：CEO 想尽快上 Hyperliquid

**场景描述**：Somi Solana 版上线 3 月，日活 200+。CEO 在会上看到 Hyperliquid 增长数据（日交易量 $4B+），要求下月开始做 Hyperliquid 版本。你对 Hyperliquid 有基本了解（自建 L1、HyperBFT、双层架构 [E05]）但没开发过。

**关键信号**：CEO 决策基于市场热度非技术评估；Hyperliquid 自建 L1 与 Solana 开发范式完全不同；1 人开双线 = 两边都做不好；需要预期管理。

**决策选项**：A. 按要求下月开始（边学边做）。B. 花 2 周做技术评估出 ROI 报告。C. 先做 EVM 兼容链（Base）再到 Hyperliquid。

**判断逻辑**：专家选 B，结论可能指向 A/C/"暂时不做"。HyperCore 是私有引擎，HyperEVM 才对外开放，需评估成熟度（职责 1.3）；2 周产出：devnet 实验报告+工程量对比+用户价值评估（知识 3.5, 3.19）；向 CEO："两周后给你带数据的建议"。

**常见陷阱**：直接说"做不到"；不做评估直接开工；评估只看技术不看业务。

**→ 对应**：职责 1.3, 6.2 / 能力 2.1, 2.4, 2.11 / 知识 3.5, 3.19 | **→ 相关场景**：6.1

## 二、策略层场景（3 个）

### 场景 2.1 优先级争论：CEO 要前端 Demo，你想先搭数据管道

**场景描述**：MVP 第 3 周。CEO 想先有"能给投资人看的前端 Demo"。你认为应先搭链上数据管道——没实时数据前端只是静态页。本周 Sprint 只能选一个。

**关键信号**：CEO 有融资时间压力（Demo 有外部价值）；数据管道是前端依赖但 mock 也能做 demo；数据管道能同时服务前端和交易引擎（杠杆高）。

**决策选项**：A. 先做前端 mock demo（1 周）。B. 先搭数据管道（1+1 周）。C. 并行：3 天最简管道（只取价格）+ 4 天用真实数据做前端。

**判断逻辑**：专家选 C。不需要完整管道——实时价格一个数据点就让 demo"活"起来（职责 2.1）；Jupiter Price API + WebSocket 3 天能通（知识 3.7）；这条最小管道是完整管道的子集不浪费（能力 2.3）。向 CEO："不是二选一，我能用更少时间把两个都做出来"。

**常见陷阱**：与 CEO 对抗"你不懂技术"；完全妥协做纯 mock 被投资人一问穿帮。

**→ 对应**：职责 2.1 / 能力 2.1, 2.11 / 知识 3.7 | **→ 相关场景**：1.1, 3.4

---

### 场景 2.2 技术债决策：关键模块需重构但营销活动在即

**场景描述**：上线 6 周。交易执行模块串行处理，并发 > 20 笔/秒时丢单。CEO 下周推营销（预计日活从 50 涨到 300，峰值可能 50+ 笔/秒）。重构需 5 天，CEO 不想推迟。

**关键信号**：丢单 = 用户资金风险不只是体验问题 [E02]；300 日活峰值超硬上限；5 天重构 vs 推迟营销的机会成本。

**决策选项**：A. 推迟营销 1 周先重构。B. 不重构，加并发限流器+排队（2 天）。C. 营销按时推，白天修 bug 晚上重构。

**判断逻辑**：专家选 B 改进版——限流器+排队+把营销推迟 3 天（而非 1 周）。丢单不可接受（职责 2.2）；完整重构 5 天可能引入新 bug（能力 2.3）；限流器 2 天止血（知识 3.3）；向 CEO："推迟 3 天不是做不到，是丢单比晚 3 天后果严重 100 倍"。营销后立即 P0 重构。

**常见陷阱**：选 C 高估精力两边都做不好；不告诉 CEO 风险直到系统崩溃。

**→ 对应**：职责 2.2 / 能力 2.3, 2.11 / 知识 3.3 | **→ 相关场景**：3.4, 4.2

### 场景 2.3 外部依赖风险：Jupiter API 突然限流

**场景描述**：周三下午热门代币上线，交易量暴涨。Jupiter Quote API 响应从 200ms 飙到 3 秒，部分 429。Somi 所有交易依赖 Jupiter——用户无法交易。CEO 问"多久能修好"。

**关键信号**：问题在 Jupiter 端我们无法直接修复；所有依赖 Jupiter 的平台都受影响；用户正在流失；无 fallback 方案暴露单点依赖。

**决策选项**：A. 等 Jupiter 恢复+发公告。B. 紧急集成 Raydium SDK 作 fallback（4-6 小时）。C. 降级服务：只支持 SOL/USDC 交易对（1 小时）。

**判断逻辑**：专家选 C 即时止血 + A 沟通 + B 作中期 action item。紧急集成 4-6 小时太冒险（能力 2.1）；降级到 SOL/USDC 1 小时内完成（职责 2.3）；发公告"部分交易对暂时不可用，核心正常"；写 P1 action item 做多 DEX fallback 架构。

**常见陷阱**：恐慌中紧急集成未测试代码；只等不动用户记住你什么都没做；事后不做 fallback 建设。

**→ 对应**：职责 2.3 / 能力 2.1, 2.11 / 知识 3.13, 3.7 | **→ 相关场景**：3.4, 4.3

## 三、执行层场景（5 个）

### 场景 3.1 Solana 合约设计：高频下的 Account Contention

**场景描述**：设计用户保证金合约。初版所有保证金存入全局 vault 账户。devnet 测试 10+ 用户同时存入时大量 tx 因 account contention 失败——Solana 并行执行要求写入不同账户的 tx 才能并行 [E09]。

**关键信号**：Sealevel 并行模型的根本约束；全局 vault 是串行化瓶颈；用户数会增长问题只会恶化；需要重新设计账户结构。

**决策选项**：A. 每用户独立 vault 账户（PDA）——完全消除 contention。B. 分片 vault（4-8 个）——随机分配用户到不同分片。C. 保持全局 vault + 前端排队——用软件层面避免并发。

**判断逻辑**：专家选 A。每用户一个 PDA vault 从根本上消除 contention（知识 3.6）；PDA 用 user pubkey 作 seed 保证确定性和唯一性（能力 2.5）；虽然增加了账户管理复杂度（结算时需遍历用户账户），但 0-1 阶段用户量小，这个代价可控。分片方案 B 在用户量极大时才有优势，现在过度设计。

**常见陷阱**：用 C 把问题推到前端——前端排队在高并发下也不可靠；选 B 过度设计增加不必要复杂度。

**→ 对应**：职责 3.1 / 能力 2.5 / 知识 3.5, 3.6, 3.10

---

### 场景 3.2 数据管道：区块重组导致数据不一致

**场景描述**：Somi 链上数据管道已运行 2 周。某天用户反馈"我的交易在页面上出现又消失了"。排查发现：Solana 发生了一次 slot 回滚（reorg），你的索引服务处理了被回滚的区块里的交易，但没有在 reorg 后撤销它们——导致"幽灵交易"显示在用户界面上。

**关键信号**：Solana 虽然最终确认快但仍有 reorg 可能；当前索引服务只有"写入"没有"回滚"逻辑；这不是一次性问题——reorg 会持续发生；用户看到"消失的交易"会严重损害信任。

**决策选项**：A. 只展示 finalized 确认级别的交易（延迟增加 ~15 秒）。B. 展示 confirmed 级别但加 reorg 回滚逻辑。C. 展示 processed 级别（最快）+ 前端标注"待确认"状态。

**判断逻辑**：专家选 B+C 组合。交易列表用 confirmed 级别 + reorg 回滚逻辑保证最终一致性（能力 2.8）；前端在交易未 finalized 前显示"确认中"标签（知识 3.7）；实现方式：索引服务记录每个 slot 的处理状态，监听 slot 确认级别变化，confirmed → finalized 时标记为最终，如果 slot 被回滚则删除对应记录。纯 finalized（A）延迟太高影响用户体验，纯 processed（C 不加回滚）会重复当前问题。

**常见陷阱**：简单选 A 牺牲体验；忽略 reorg 频率认为"很少发生"；没有在 UI 层面做状态区分。

**→ 对应**：职责 3.3 / 能力 2.8 / 知识 3.5, 3.7 | **→ 相关场景**：3.4

---

### 场景 3.3 前端交易 UX：签名后到确认前的中间态

**场景描述**：用户在 Somi 点击"交易"后，需要在 Phantom 钱包签名→ tx 提交到 Solana → 等待确认。这个过程通常 1-3 秒，但网络拥堵时可能 10-30 秒甚至失败。当前 UI 在签名后只显示一个 loading spinner，用户不知道发生了什么。已有 3 个用户反馈"以为交易没成功就重复提交了"。

**关键信号**：重复提交 = 重复交易 = 用户资金多扣 → 严重问题；1-3 秒用户能等，10+ 秒用户会恐慌；当前 UI 没有任何状态反馈。

**决策选项**：A. 签名后禁用按钮+显示进度条。B. 签名后显示 tx hash 和 explorer 链接+轮询确认状态。C. 签名后立即在前端模拟成功（optimistic UI），后台确认后更新。

**判断逻辑**：专家选 B 为主体 + A 的防重复提交。具体：(1) 签名后立即禁用交易按钮防止重复提交；(2) 显示 tx hash（可点击跳转 Solscan）让用户知道"交易已提交，区块链正在处理"；(3) 后台用 confirmTransaction 轮询状态，成功/失败/超时分别更新 UI（能力 2.7、知识 3.16）。不选 C 因为交易场景不适合 optimistic UI——如果后台确认失败但前端已显示成功，用户会以为资金已到位做出后续操作。

**常见陷阱**：选 C（optimistic UI）在交易场景是危险的；只加 loading 不加 tx hash 用户无法自行验证；不防重复提交。

**→ 对应**：职责 3.4 / 能力 2.7 / 知识 3.16, 3.15

### 场景 3.4 交易执行：极端行情下着陆率暴跌

**场景描述**：某天 BTC 剧烈波动，Solana 网络拥堵。Somi 的交易着陆率从平时的 95% 降到 40%——60% 的交易因为 blockhash 过期或 compute budget 不足而失败。用户抱怨"交易一直失败"。

**关键信号**：blockhash 有效期约 60 秒 → 网络拥堵时 tx 在 mempool 排队超时 [E07]；compute budget 默认值可能不够 → 复杂 swap 路径需要更多 CU；priority fee 太低 → 在拥堵时 validator 优先处理高 fee 的 tx。

**决策选项**：A. 提高 priority fee 10 倍（成本增加但着陆率提升）。B. 实现自动重试+动态 priority fee 调整。C. 切换到 Jito bundle 提交通道。

**判断逻辑**：专家选 B+C 组合。(1) 实现动态 priority fee：基于最近 5 个 slot 的 fee 分布取 P75 值作为基准，拥堵时自动上浮（能力 2.9、知识 3.15）；(2) 自动重试：tx 失败后获取新 blockhash 重新构造并提交，最多 3 次；(3) 对高价值交易走 Jito bundle 通道——bundle 内 tx 要么全部成功要么全部失败，且有 MEV 保护（知识 3.15）。不能只靠 A（固定高 fee）——平时浪费成本，极端时可能还不够。

**常见陷阱**：暴力提高 fee 不做动态调整（浪费）；无限重试不设上限（重复交易风险）；不区分普通交易和高价值交易一刀切。

**→ 对应**：职责 3.6, 4.2 / 能力 2.9 / 知识 3.15, 3.5 | **→ 相关场景**：2.2, 2.3

### 场景 3.5 DevOps 决策：RPC 节点方案选择

**场景描述**：Somi 目前用 Helius 免费 tier 的 RPC 服务（50 req/s）。随着用户增长，高峰期开始触及速率限制。你需要决定下一步：升级到 Helius 付费版（$50/月，200 req/s）、切换到 QuickNode（$100/月，更高性能）、还是自建 Solana RPC 节点（$500/月服务器成本+运维时间）。

**关键信号**：当前瓶颈是 50 req/s → 付费版 200 req/s 能覆盖近 2-3 个月增长；自建节点有 Geyser 插件优势（最低延迟数据流 [E07]）但运维成本高；0-1 阶段工程师时间比服务器成本贵。

**决策选项**：A. 升级 Helius 付费版（$50/月）。B. 自建 RPC 节点（$500/月+运维时间）。C. Helius 付费版+预留 Geyser 切换接口。

**判断逻辑**：专家选 C。$50/月解决眼前问题，不值得花工程时间自建节点（能力 2.1）；但代码层面抽象出 DataProvider 接口，未来需要 Geyser 低延迟数据流时可以平滑切换（职责 3.5、知识 3.7）；自建节点留到阶段二交易量真正需要亚秒级数据时再考虑。

**常见陷阱**：直接自建节点（过度工程化）；不做接口抽象硬编码 Helius（未来迁移成本高）。

**→ 对应**：职责 3.5 / 能力 2.10, 2.1 / 知识 3.7, 3.4

## 四、专业深度层场景（3 个）

### 场景 4.1 安全事件：上线后发现潜在漏洞

**场景描述**：Somi vault 合约上线 2 周，TVL $8K。你在审查代码时发现一个潜在漏洞：withdraw 函数在更新余额之前发送了 SOL——经典的重入攻击模式 [E08]。虽然 Solana 的执行模型（单线程 CPI）让传统重入攻击更难实现，但你不能 100% 确定所有攻击路径都被封堵。

**关键信号**：TVL $8K 是用户真金白银；Solana 的 CPI 模型降低了重入风险但不消除 [E10]；合约有 pause 功能（场景 1.2 的决策果实）；修复需要部署新合约版本。

**决策选项**：A. 立即 pause 合约，通知用户，修复后重新部署。B. 先修复+测试+部署新版本，测试通过后迁移用户资金。C. 评估实际攻击可行性，如果攻击难度极高则不暂停，静默修复。

**判断逻辑**：专家选 A。在加密交易领域，涉及资金的潜在漏洞不赌概率（职责 4.1、能力 2.2）；pause 是场景 1.2 中部署的安全网，现在正是用它的时候；暂停后：(1) 24 小时内完成修复+测试+外部快速审查；(2) 部署新版本到 devnet 验证；(3) 主网部署新版本，用户迁移资金到新合约。$8K TVL 暂停 24 小时的业务损失远小于被攻击的后果。

**常见陷阱**：选 C 心存侥幸——"攻击难度高"不等于"不会被攻击"；选 B 但迁移期间旧合约仍然暴露；修复后不做回归测试。

**→ 对应**：职责 4.1, 1.2 / 能力 2.2 / 知识 3.9, 3.10 | **→ 相关场景**：1.2

### 场景 4.2 延迟优化：竞品交易确认比我们快

**场景描述**：用户反馈"在 X 平台交易 1 秒就确认了，在 Somi 要 3-5 秒"。你分析后发现：Somi 的交易流程是 前端构造 tx → 发送到公共 RPC → 等待确认，没有做任何优化。竞品可能在用 Jito bundle 或私有 RPC 节点。

**关键信号**：3-5 秒 vs 1 秒在交易场景是显著差异；公共 RPC 的延迟和着陆率都不如付费/私有节点 [E07]；Jito bundle 不只是更快还有 MEV 保护。

**决策选项**：A. 升级到付费 RPC（如 Helius 专用节点）。B. 集成 Jito SDK 使用 bundle 提交。C. 两者都做 + 交易预构造（预签名+预取 blockhash）。

**判断逻辑**：专家选 C 但分阶段。第一步（2 天）：升级到付费 RPC——最快见效（职责 4.2）；第二步（3 天）：集成 Jito bundle——不只是快还有 MEV 保护（知识 3.15）；第三步（2 天）：实现交易预构造——用户点击"交易"时 blockhash 已预取、CU 已估算、指令已组装，只等签名后立即发送（能力 2.9）。三步合计可以把延迟从 3-5 秒降到 < 1 秒。

**常见陷阱**：只做 A 不做 B/C（优化不到位）；一次性全部做（拖太长时间用户已经流失）。

**→ 对应**：职责 4.2 / 能力 2.9 / 知识 3.15, 3.5 | **→ 相关场景**：3.4, 2.2

### 场景 4.3 DeFi 集成：多 DEX 智能路由设计

**场景描述**：Somi 当前只接 Jupiter swap，但某些长尾代币对 Jupiter 路由不到最优价格（因为流动性分散在 Raydium/Orca/Meteora 多个池子）。CEO 要求"给用户最好的价格"。你需要决定是继续依赖 Jupiter 还是自建路由层。

**关键信号**：Jupiter 本身就是聚合器已经做了路由优化 [E05]；长尾代币流动性确实可能分散；自建路由层 = 重造轮子，ROI 存疑；可能是 Jupiter API 参数没调优而非路由不够好。

**决策选项**：A. 自建多 DEX 路由聚合层。B. 优化 Jupiter API 调用参数（slippage、route 数量限制）。C. Jupiter 为主 + 对特定高价值交易对直连 Raydium/Orca SDK。

**判断逻辑**：专家选 B 先行，必要时加 C。先排查是不是 Jupiter 参数没调好——可能只是 maxAccounts 限制太低或 slippage 设置不当（能力 2.4、知识 3.13）；如果调参后仍然对特定交易对不优，再针对性直连底层 DEX（职责 4.3）。自建路由层（A）在 0-1 阶段完全不值得——Jupiter 的路由算法有几十人在优化，一个人不可能做得更好。

**常见陷阱**：不调参就下结论"Jupiter 不够好"；为了"最好的价格"花 3 周自建路由层（机会成本远大于价差收益）。

**→ 对应**：职责 4.3, 2.3 / 能力 2.4, 2.9 / 知识 3.13, 3.11 | **→ 相关场景**：1.1, 2.3

## 五、组织层场景（3 个）

### 场景 5.1 技术文档：新工程师下周入职但什么都没写

**场景描述**：好消息——CEO 融到了资，下周一有一位 junior 工程师入职。坏消息——Somi 的整个技术栈只在你脑子里：没有架构文档、没有部署手册、合约交互逻辑只有代码注释。你这周还有正常的开发任务。新人入职第一天，你希望他能自己搭好开发环境并理解系统全貌。

**关键信号**：所有知识在你一个人脑子里 = 极高的 bus factor 风险 [E02]；下周一是硬 deadline；不可能一周内写完所有文档；需要优先级排序。

**决策选项**：A. 这周全力写文档，暂停开发。B. 写一份最小的 onboarding 文档（README + 架构图 + 本地启动指南），其他靠 pair programming。C. 不写文档，第一周和新人 pair programming 手把手带。

**判断逻辑**：专家选 B。(1) 最小 onboarding 文档（1 天）：README（项目概览+技术栈+目录结构）、一张架构图（系统组件关系）、本地开发环境搭建步骤（step-by-step 可复现）（职责 5.1、能力 2.12）；(2) 这份文档的标准是"新人照着做能在 2 小时内跑通本地环境"；(3) 深层架构知识用 pair programming 传递，但每次 pair 后让新人把学到的东西写进文档——让新人成为文档的"第一个贡献者"。选 A 停止开发不现实（0-1 阶段耽误不起）；选 C 无文档完全靠口传是不负责任的。

**常见陷阱**：试图写完美的全量文档（不可能一周完成）；完全不写靠 pair（你生病一天新人就卡住）。

**→ 对应**：职责 5.1 / 能力 2.12 / 知识 3.4 | **→ 相关场景**：5.2

### 场景 5.2 跨职能沟通：CEO 想要的功能有重大技术风险

**场景描述**：CEO 看到竞品上线了"一键复制交易"功能（copy trading），要求 Somi 也做。你的技术评估：copy trading 需要代理交易权限——即 Somi 的后端需要能代替用户签名执行交易。这在技术上意味着用户需要把私钥权限委托给 Somi，带来严重的安全和信任风险。

**关键信号**：代理交易 = 托管用户资金权限 → 监管和安全的雷区 [E02]；竞品可能用了不同的技术方案（如 session key / delegated signing）；CEO 看到的是产品机会，你看到的是技术风险。

**决策选项**：A. 直接拒绝"做不了"。B. 调研 session key / Solana Blinks 等非托管方案，给 CEO 一个替代方案。C. 按 CEO 要求做标准的托管式 copy trading。

**判断逻辑**：专家选 B。(1) 不说"做不到"——说"这个功能有一个更安全的实现方式"（能力 2.11、职责 5.3）；(2) 调研 Solana 的 delegated signing（session keys / 委托授权机制），用户授权有限的交易权限而非完整私钥——既满足 copy trading 需求又不触碰资金托管红线；(3) 3 天内给 CEO 一份方案对比报告：托管式（风险+合规成本）vs 非托管式（功能限制+用户体验权衡）。

**常见陷阱**：直接拒绝（错失产品机会）；不调研直接做托管方案（埋下安全和合规炸弹）。

**→ 对应**：职责 5.3, 4.4 / 能力 2.11, 2.4 / 知识 3.9, 3.21 | **→ 相关场景**：1.2

### 场景 5.3 开源决策：是否开源核心交易模块

**场景描述**：CEO 想通过开源吸引社区开发者和建立技术品牌。问题是：Somi 的核心价值之一是交易执行引擎的优化（延迟、着陆率、路由策略）。开源这部分代码可能让竞品直接复制你的优势。

**关键信号**：Web3 社区重视开源 → 不开源会影响品牌和招聘 [E04, E12]；核心交易优化是竞争壁垒 → 开源有商业风险；可以选择性开源（前端/工具库开源，核心引擎不开源）。

**决策选项**：A. 全部开源。B. 选择性开源：前端 UI + SDK 开源，核心引擎闭源。C. 全部闭源。

**判断逻辑**：专家选 B。(1) 前端 UI 和 SDK 开源——降低集成门槛、吸引社区贡献、建立技术品牌（职责 5.2）；(2) 核心交易引擎闭源——这是 Somi 的竞争壁垒（知识 3.20）；(3) 开源部分要维护好——README 完整、issue 及时响应、CI/CD 通过——半吊子开源比不开源更伤品牌（能力 2.13）。

**常见陷阱**：全部开源失去竞争壁垒；全部闭源在 Web3 世界里失去信任和社区。

**→ 对应**：职责 5.2 / 能力 2.13 / 知识 3.18, 3.22

## 六、自进化层场景（3 个）

### 场景 6.1 前沿评估：Firedancer 升级对交易引擎的影响

**场景描述**：Solana Foundation 宣布 Firedancer（由 Jump Crypto 开发的独立验证者客户端）将在下季度上主网。Firedancer 号称将 Solana 吞吐量提升 10 倍以上 [E09]。CEO 问你"这对我们有什么影响"。

**关键信号**：吞吐量提升 → 网络拥堵减少 → 交易着陆率自然提升；独立客户端 → 验证者多样性增加 → 网络更稳定；但也可能带来客户端行为差异 → 交易确认时间特征可能变化。

**决策选项**：A. 持续关注但不采取行动。B. 在 Firedancer testnet 上测试 Somi 交易流程。C. 基于 Firedancer 的新特性重新优化交易引擎。

**判断逻辑**：专家选 B 先行。(1) 不能等到上主网才测——提前在 testnet 验证 Somi 的交易流程在 Firedancer 下是否正常（职责 6.1）；(2) 重点测试：交易确认时间分布、priority fee 行为、Geyser 接口兼容性（能力 2.14）；(3) 测试结果产出一份评估报告："无影响/需要微调/需要重大改动"，这是给 CEO 的回答；(4) 如果测试发现 Firedancer 的高吞吐量使得 priority fee 动态调整策略需要更新（场景 3.4 的延续），提前准备。

**常见陷阱**：忽略不管（上主网当天出问题才知道）；过度反应（还没上主网就开始大改架构）。

**→ 对应**：职责 6.1 / 能力 2.14 / 知识 3.5, 3.9 | **→ 相关场景**：3.4, 1.3

### 场景 6.2 多链储备：从 Solana 到 EVM 的技术切换准备

**场景描述**：Somi 阶段二计划扩展到 BSC 和 Base。CEO 问"需要提前准备什么"。当前代码库完全围绕 Solana 生态（Anchor/Rust/@solana/web3.js），EVM 相关代码为零。

**关键信号**：Solana → EVM 不只是换 SDK——账户模型、Gas 模型、合约语言完全不同 [E06]；但 Web 后端和前端框架可以复用；现在不需要写 EVM 代码但需要在架构上为多链做准备。

**决策选项**：A. 现在就开始写 EVM 代码（并行开发）。B. 在架构层面做抽象（ChainAdapter 接口），但不写 EVM 实现。C. 等到阶段二开始时再考虑。

**判断逻辑**：专家选 B。(1) 不需要现在写 EVM 代码——0-1 阶段 Solana 还没做好（职责 6.2）；(2) 但在关键模块抽象出 ChainAdapter 接口：signTransaction()、submitTransaction()、getBalance()、subscribeEvents()——未来实现 EVMAdapter 时只需要填充这些方法（能力 2.1、知识 3.19）；(3) 个人学习层面：每月花 4 小时在 Foundry 上写一个小合约保持 Solidity 手感（能力 2.14）。

**常见陷阱**：选 A 分散精力；选 C 等到要做时发现代码耦合严重重构成本巨大。

**→ 对应**：职责 6.2 / 能力 2.1, 2.14 / 知识 3.8, 3.19 | **→ 相关场景**：1.3

### 场景 6.3 方法论沉淀：把踩过的坑变成团队资产

**场景描述**：Somi 上线 4 个月，你已经踩过：Solana account contention（场景 3.1）、区块重组数据不一致（场景 3.2）、Jupiter API 限流（场景 2.3）、交易着陆率暴跌（场景 3.4）等十几个坑。第二个工程师已入职 2 周。你发现他开始踩你踩过的坑——他不知道"为什么 vault 账户要用 PDA 而不是全局"。

**关键信号**：知识只在你脑子里 → 团队不会自动学到你的经验；口头交代不可靠——两周后就忘了；每次踩同样的坑 = 团队效率浪费。

**决策选项**：A. 花一周写完整的技术知识库。B. 每次新人踩坑时再口头解释。C. 建立"踩坑→复盘→写 playbook"的持续机制。

**判断逻辑**：专家选 C。(1) 不是一次性写完——是建立一个持续运转的机制（职责 6.3）；(2) 每次故障或重大技术决策后，30 分钟内写一份简短的 postmortem/decision record：问题是什么→为什么会发生→我们做了什么→下次怎么避免（能力 2.15）；(3) 让新工程师参与写——"你刚踩的这个坑，请写一份 300 字的 TIL（Today I Learned）到 wiki 里"；(4) 这些 TIL 积累起来就是团队的知识资产（知识 3.4）。

**常见陷阱**：选 A 试图一次写完（不现实且很快过时）；选 B 完全靠口传（新人不敢问、你会遗漏）。

**→ 对应**：职责 6.3 / 能力 2.15, 2.12 / 知识 3.4 | **→ 相关场景**：5.1
