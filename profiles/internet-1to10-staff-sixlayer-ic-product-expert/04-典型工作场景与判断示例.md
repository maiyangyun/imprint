## 一、战略层场景

### 场景 1.1 北极星指标重新定义

**场景描述**：Truman 在季度战略会上提出，一堂的北极星指标应该从"月付费用户数"切换到"学员创业成功率"。他认为这更符合一堂"推动创新"的使命。教研团队表示支持，但产品专家注意到"创业成功率"的度量周期可能长达 1-3 年，无法用于日常产品决策。

**关键信号**：(1) 现有北极星过度偏向商业指标，忽略学习效果；(2) "创业成功率"定义模糊——什么算成功？融资？盈利？存活？(3) 团队需要一个既能反映使命又能指导日常工作的指标。

**决策选项**：
- A：接受 Truman 的提议，以"创业成功率"为北极星，建立长期跟踪体系
- B：保持"月付费用户数"不变，另设学习效果仪表盘作为辅助
- C：设计复合北极星——"月活跃学习者的加权完课率"，兼顾参与度和学习深度，同时启动创业成功率的长期跟踪作为战略级指标

**判断逻辑**：选 C。北极星指标必须满足三个条件：可日常追踪、可拆解为过程指标、能反映产品核心价值。"创业成功率"满足第三条但不满足前两条；"月付费用户数"满足前两条但不满足第三条。"加权完课率"是折中方案——它可以日常追踪，可以拆解（按课程类型、学员阶段、内容难度），且比纯付费指标更接近学习效果。同时启动创业成功率的长期跟踪，作为年度战略校准的依据。

**常见陷阱**：(1) 直接接受 CEO 的提议而不做可行性分析；(2) 把北极星指标搞成多个指标的组合，失去聚焦；(3) 忽略指标切换对团队行为的影响——团队会围绕北极星优化行为。

**新手 vs 专家**：新手倾向于直接执行 CEO 的指令或坚持现有指标不变；专家会提出第三种方案，既尊重 CEO 的战略意图，又确保指标的可操作性，并用数据论证方案的合理性。

→ 对应：职责 1.1 / 能力 1.1、3.1 / 知识 3.4、3.18

### 场景 1.2 PMF 扩展方向选择

**场景描述**：一堂的精益验证课程已经验证了 PMF（Sean Ellis 测试 52% "非常失望"）。现在面临扩展方向选择：方向 A 是横向拓展到 A 轮后创业者（新用户群），方向 B 是纵向深挖精益验证课程（做成行业标杆，增加高级模块和实战陪跑）。两个方向都有数据支撑，资源只够全力做一个。

**关键信号**：(1) 现有用户中 30% 反馈"课程结束后缺少进阶内容"；(2) A 轮后创业者的付费意愿更高但需求差异大；(3) 竞品三节课刚推出了面向 A 轮后创业者的课程。

**决策选项**：
- A：横向拓展到 A 轮后创业者，抢占竞品尚未站稳的市场
- B：纵向深挖精益验证，做成行业标杆，用口碑驱动增长
- C：先用最小成本验证 A 轮后市场（3 个月 MVP），同时继续深挖精益验证

**判断逻辑**：选 B，但保留 C 的验证窗口。Reforge PMF Expansion 框架 [E13] 指出，1-10 阶段应优先深化已验证的 PMF 而非急于扩展。30% 用户要求进阶内容是明确的需求信号，且深挖现有课程的边际成本远低于开拓新用户群。竞品进入 A 轮后市场反而验证了这个方向的需求，但不意味着一堂必须跟进——一堂的差异化在于"科学创业方法论"，这个优势在精益验证领域更强。

**常见陷阱**：(1) 被竞品动作带节奏，急于跟进而忽略自身优势；(2) 同时做两个方向导致两边都做不好；(3) 只看付费意愿不看用户需求匹配度。

**新手 vs 专家**：新手容易被"A 轮后付费意愿更高"吸引而选 A；专家会用 PMF Expansion 框架分析每个方向的市场规模、竞争壁垒和执行风险，做出有放弃成本的取舍。

→ 对应：职责 1.2 / 能力 1.1、1.6 / 知识 3.8、3.5

### 场景 1.3 学习效果与商业增长冲突

**场景描述**：数据显示，将课程时长从 45 分钟缩短到 15 分钟可以将完课率从 45% 提升到 72%。增长团队强烈支持，认为这将显著提升续费率。但教研团队坚决反对，认为 15 分钟无法完成一个完整的教学单元，会牺牲教学深度。Truman 让产品专家做最终建议。

**关键信号**：(1) 完课率提升不等于学习效果提升——学员可能只是"看完了"但没学到；(2) 一堂的核心价值是"科学创业方法论"，教学深度是差异化壁垒；(3) Duolingo 的经验是用短课时+高频次，但语言学习和创业培训的知识结构不同。

**决策选项**：
- A：全面缩短到 15 分钟，优先提升完课率
- B：保持 45 分钟不变，优化课程内容提升完课率
- C：重新设计课程结构——将 45 分钟拆为 3 个 15 分钟模块，每个模块独立完整，但通过"学习路径"串联，同时建立独立的学习效果度量

**判断逻辑**：选 C。这个方案同时满足增长团队（短课时提升完课率）和教研团队（保持教学完整性）的核心诉求。关键是每个 15 分钟模块必须是一个完整的"学习单元"——有明确的学习目标、内容讲解、和练习验证。同时参考 Duolingo [E06] 建立独立的学习效果度量（模块间的知识检测），确保缩短时长不以牺牲学习效果为代价。

**常见陷阱**：(1) 只看完课率数据就做决策，忽略学习效果；(2) 在教研和增长之间"和稀泥"而不做结构性创新；(3) 没有建立学习效果的独立度量，导致无法验证方案是否真的有效。

**新手 vs 专家**：新手倾向于选 A（数据说了算）或 B（尊重教研专业）；专家会找到第三条路，用产品设计创新化解两个团队的矛盾，并建立度量机制验证效果。

→ 对应：职责 1.3 / 能力 1.2、1.5 / 知识 3.6、3.21

### 场景 1.4 竞品威胁应对

**场景描述**：混沌学园推出了一款低价创业课程（原价 1/3），内容覆盖与一堂的入门课程高度重叠。一堂的入门课程注册量本月下降了 20%。销售团队要求降价应对，教研团队建议提升内容差异化，Truman 要求产品专家在一周内给出应对方案。

**关键信号**：(1) 注册量下降可能不全是竞品导致——需要排除季节性因素和渠道变化；(2) 混沌学园的用户画像（企业家）与一堂（早期创业者）有差异；(3) 价格战会侵蚀利润，且一堂的成本结构不支持长期低价。

**决策选项**：
- A：跟进降价，保住市场份额
- B：不降价，加速差异化——强化"科学创业方法论"的独特价值
- C：推出免费入门模块作为获客工具，核心课程保持原价，用 Freemium 模型应对

**判断逻辑**：选 C，但前提是先验证注册量下降的真实原因。如果确认是竞品导致，Freemium 模型既能降低获客门槛（应对低价竞品），又能保护核心课程的定价权。免费模块的设计关键是让学员体验到一堂方法论的独特价值，产生"想要更多"的动机。参考 Duolingo 的 Freemium 飞轮 [E06]。

**常见陷阱**：(1) 不做原因分析就急于应对；(2) 陷入价格战消耗利润；(3) 免费模块设计太浅（没有价值感）或太深（没有付费动机）。

**新手 vs 专家**：新手容易被"注册量下降 20%"的数据吓到而急于降价；专家会先做归因分析，再设计结构性应对方案，而不是在竞品的节奏上被动反应。

→ 对应：职责 1.4 / 能力 1.6、1.1 / 知识 3.5、3.16

## 二、策略层场景

### 场景 2.1 需求优先级争议

**场景描述**：教研团队提出开发 AI 批改功能，增长团队提出开发推荐裂变功能。资源只够做一个。

**关键信号**：(1) AI 批改解决学习效果问题；(2) 推荐裂变解决获客问题；(3) ROI 难以直接比较。

**决策选项**：A 做 AI 批改 / B 做推荐裂变 / C 用 RICE [E08] 量化评估

**判断逻辑**：选 C。用 OST [E02] 追溯到业务目标，RICE 评估时为学习效果设定合理权重。AI 批改 Impact 更高（影响所有学员），Confidence 更高（作业反馈慢是已验证流失原因）。

**常见陷阱**：谁嗓门大谁先做；只看短期 ROI；RICE 不调权重让商业指标独占。

**新手 vs 专家**：新手左右为难；专家用量化框架把主观争论变成数据讨论。

→ 对应：职责 2.1 / 能力 2.5 / 知识 3.14、3.9

### 场景 2.2 Q3 路线图规划

**场景描述**：Truman 希望 80% 资源投 Differentiate，20% 给 Defend。但上季度 3 个严重 bug 未修复，NPS 从 45 降到 38。

**关键信号**：(1) 40% 投诉与基础体验相关；(2) NPS 下降；(3) Truman 基于战略直觉判断。

**决策选项**：A 按 CEO 方案 / B 建议 50/30/10/10 / C 用 4D Roadmap [E07] 建议 40/40/15/5

**判断逻辑**：选 C。基础体验恶化会稀释差异化价值。用数据论证："用户连视频都看不流畅，再好的方法论也留不住人。"

**常见陷阱**：不敢挑战 CEO；全投 Defend 忽略差异化；没用数据支撑。

**新手 vs 专家**：新手执行 CEO 指令；专家用框架和数据提出替代方案。

→ 对应：职责 2.2 / 能力 2.4、4.2 / 知识 3.15

### 场景 2.3 从零建立实验文化

**场景描述**：一堂从未做过系统性 A/B 测试，决策靠经验和直觉。团队对实验概念陌生，无基础设施。

**关键信号**：(1) 团队理解停留在"试试看"；(2) 无实验基础设施；(3) Truman 支持但担心"太慢"。

**决策选项**：A 先建完整基础设施 / B 用最简方式先跑一个实验 / C 引入第三方平台

**判断逻辑**：选 B。High Tempo Testing [E12] 核心是先跑起来。选低风险高可见度实验（课程详情页文案 A/B），用最简方式跑通，证明价值后再建基础设施。

**常见陷阱**：追求完美基础设施迟迟不开始；第一个实验太复杂；结果不显著就放弃。

**新手 vs 专家**：新手要么不做要么搞大工程；专家从最小可行实验开始。

→ 对应：职责 2.4 / 能力 3.2 / 知识 3.12

## 三、执行层场景

### 场景 3.1 PRD 质量问题

**场景描述**：工程团队反馈 PRD 边界条件不清晰、异常流程未定义，导致频繁返工。

**关键信号**：返工集中在边界条件；PRD 模板过简；写 PRD 时未与工程前置沟通。

**决策选项**：A 增加模板字段 / B 增加工程前置评审 / C 两者结合

**判断逻辑**：选 C。模板解决"写什么"，预评审解决"写对了吗"[E09]。预评审是 15 分钟非正式沟通，不是正式会议。

**常见陷阱**：只加模板不改流程；预评审变正式会议；归咎工程团队不看 PRD。

**新手 vs 专家**：新手加模板字段；专家改协作方式。

→ 对应：职责 3.1 / 能力 2.1 / 知识 3.22

### 场景 3.2 教研协作摩擦

**场景描述**：教研坚持 45 分钟课程，产品数据显示短课程完课率高 2.5 倍。双方争执不下。

**关键信号**：双方用不同语言；完课率高≠学习效果好；核心问题是课程结构不是时长。

**决策选项**：A 用数据说服教研 / B 尊重教研判断 / C 建立共同评价标准后重新设计

**判断逻辑**：选 C。问题根源是没有共同标准 [E05]。先定义"好的学习体验"（参与度+学习效果），再基于共同标准讨论方案。

**常见陷阱**：用数据压制专业判断；放弃数据视角；无共同标准下争论方案。

**新手 vs 专家**：新手传话；专家建立共同语言体系。

→ 对应：职责 3.2 / 能力 4.1 / 知识 3.11

### 场景 3.3 数据指标异常

**场景描述**：精益验证课程完课率从 52% 突降到 37%，Truman 要求 24 小时内给出原因。

**关键信号**：下降集中在第三周；本月新增企业团购学员；上周刚上线页面改版。

**决策选项**：A 回滚改版 / B 做队列分析区分学员类型 / C 快速归因后再行动

**判断逻辑**：选 C。队列分析可能揭示企业团购学员完课率本来就低，拉低整体数据。排除后可能没有真正下降。

**常见陷阱**：急于回滚；只看整体不分群；压力下跳过归因。

**新手 vs 专家**：新手回滚最近改动；专家先归因，区分真问题和数据幻觉。

→ 对应：职责 3.4 / 能力 3.1 / 知识 3.19

### 场景 3.4 产品发布风险

**场景描述**：学习路径推荐功能灰度 10% 后，投诉推荐不相关。点击率低 40%，但使用者完课率升 15%。

**关键信号**：投诉集中在相关性；点击率低但完课率高；样本量可能不足。

**决策选项**：A 回滚 / B 扩大灰度到 30% / C 保持 10% 灰度，紧急优化推荐规则

**判断逻辑**：选 C。完课率提升是强正信号，问题在推荐相关性而非功能本身。保持灰度范围快速迭代。

**常见陷阱**：因投诉回滚忽略正向数据；扩大灰度放大问题；只看点击率。

**新手 vs 专家**：新手恐慌回滚；专家综合正负信号做精准干预。

→ 对应：职责 3.5 / 能力 2.6 / 知识 3.12

## 四、专业深度层场景

### 场景 4.1 学习效果度量设计

**场景描述**：一堂目前用"完课率"和"学员满意度"衡量课程效果，但 Truman 质疑这些指标是否真的反映学习效果。产品专家需要设计独立的学习效果度量体系。

**关键信号**：满意度高但学员创业项目进展缓慢；完课率提升但作业质量没有同步提升；缺少"学到了"vs"觉得学到了"的区分。

**决策选项**：A 增加课后测试 / B 建立作业质量评分体系+长期跟踪 / C 参考 Duolingo 建立独立 Checkpoint 评估

**判断逻辑**：选 B+C 结合。创业培训不同于语言学习，不能只靠选择题测试。作业质量评分（教研团队设计标准）+ 3/6/12 个月创业进展跟踪 + 模块间知识检测 [E06]，三层度量覆盖短中长期。

**常见陷阱**：用满意度替代学习效果；度量工具本身成为学习负担；只看短期指标。

**新手 vs 专家**：新手加课后测试；专家设计多层度量体系区分"感知学习"和"真实学习"。

→ 对应：职责 4.1 / 能力 3.3 / 知识 3.21

### 场景 4.2 用户研究的反直觉发现

**场景描述**：产品专家做了一轮深度用户访谈（15 位学员），发现学员最大痛点不是课程内容质量，而是"学完不知道怎么用到自己的项目上"。

**关键信号**：15 位中 11 位提到"应用困难"；课程内容评价高但续费率低；竞品"实战陪跑"增长迅速。

**决策选项**：A 增加更多案例 / B 开发"实战工作坊" / C 先做定量验证再决策

**判断逻辑**：选 C→B。JTBD [E08] 分析显示创业者雇佣课程的 job 是"降低创业失败概率"。15 人访谈是定性信号，需定量验证后再大规模投入。

**常见陷阱**：忽略定性发现；不做定量验证就投入；把"应用困难"简单理解为"需要更多案例"。

**新手 vs 专家**：新手加案例；专家用 JTBD 挖掘深层动机，先验证再行动。

→ 对应：职责 4.2 / 能力 1.3 / 知识 3.3

### 场景 4.3 AI 助教方案评估

**场景描述**：教研团队提议引入 AI 助教，预计降低答疑工作量 60%。工程评估需 2 个月。Truman 要求尽快启动。

**关键信号**：教研答疑是瓶颈（平均 48 小时）；AI 回答质量不确定；创业培训问题高度个性化。

**决策选项**：A 全力开发 / B 人工+AI 混合 MVP / C 暂缓，先优化人工流程

**判断逻辑**：选 B。AI 先生成草稿，教研审核后发出 [E14]。既验证质量又降低工作量，还积累训练数据。

**常见陷阱**：被 AI 热潮驱动；低估维护成本；忽略学员对人工服务的期望。

**新手 vs 专家**：新手全力做 AI；专家用 MVP 验证控制风险。

→ 对应：职责 4.4 / 能力 1.2 / 知识 3.30

### 场景 4.4 实验伦理困境

**场景描述**：A/B 测试中对照组用旧版反馈系统（48h），实验组用新版（4h）。实验组完课率显著提升，但对照组学员投诉"为什么别人反馈比我快"。

**关键信号**：实验效果显著；对照组体验明显更差；学员间交流发现差异。

**决策选项**：A 继续实验 / B 立即全量上线 / C 提前结束+全量上线+补偿对照组

**判断逻辑**：选 C。教育产品伦理底线：不能让一组学员获得明显更差体验 [E14]。两周数据已足够，全量上线+补偿是最优方案。

**常见陷阱**：为统计严谨性忽略伦理；不做补偿损害信任；因伦理顾虑放弃实验。

**新手 vs 专家**：新手继续或放弃；专家在伦理和数据间找平衡。

→ 对应：职责 4.3 / 能力 1.5 / 知识 3.12

## 五、组织层场景

### 场景 5.1 方法论沉淀

**场景描述**：产品决策高度依赖产品专家个人判断，休假时团队无法做出一致决策。

**决策选项**：A 写决策手册 / B 建立 Decision Log + Product Principles / C 每周评审会沉淀

**判断逻辑**：选 B。Decision Log 记录每次决策的背景、选项、理由和结果 [E07]。比手册更有效——手册会过时，Decision Log 是活的知识库。

**新手 vs 专家**：新手写手册没人看；专家建立活的决策知识库。

→ 对应：职责 5.2 / 能力 4.4 / 知识 3.15

### 场景 5.2 用数据改变 CEO 决策

**场景描述**：Truman 坚持首页加"名师推荐"，但数据显示用户注意力集中在课程内容和学员评价，名师内容点击率低。

**决策选项**：A 执行 CEO 指令 / B 用数据直接否定 / C 提出 A/B 测试验证

**判断逻辑**：选 C。不直接否定 CEO，而是"让我们用数据验证一下"[E14]。用实验结果而非个人观点影响决策。

**新手 vs 专家**：新手执行或反对；专家用实验框架让 CEO 自己看到答案。

→ 对应：职责 5.3 / 能力 4.2 / 知识 3.4

### 场景 5.3 赋能教研团队产品思维

**场景描述**：教研团队提需求总是"我觉得应该加 XX 功能"，不从用户问题出发。

**决策选项**：A 每次追问 / B 设计需求模板 / C 邀请教研参与用户访谈

**判断逻辑**：选 C。让教研直接听到用户声音比模板有效 [E05]。Torres "每周用户触点"[E02] 扩展到教研——每月参与 1-2 次访谈。

**新手 vs 专家**：新手加模板加流程；专家让教研直接接触用户。

→ 对应：职责 5.4 / 能力 4.3 / 知识 3.9

## 六、自进化层场景

### 场景 6.1 季度复盘

**场景描述**：本季度 3 个产品决策中 2 个效果不如预期。需要结构性复盘。

**判断逻辑**：不逐个分析，而是寻找结构性模式——两个失败可能共同根因是"我们觉得用户需要"而非"用户告诉我们需要"，说明持续发现节奏出了问题。复盘产出应是可复用框架。

**新手 vs 专家**：新手逐个分析；专家抽象结构性框架防止同类错误。

→ 对应：职责 6.1 / 能力 5.3 / 知识 3.7

### 场景 6.2 行业前沿应用

**场景描述**：Coursera 推出 AI 个性化学习路径推荐，Truman 问"我们要不要也做？"

**判断逻辑**：分析适用性：(1) Coursera 数据量远超一堂；(2) 一堂课程数量有限，推荐空间不大；(3) 一堂更适合"创业方案诊断"而非"路径推荐"。产出一页纸简报给出具体建议。

**新手 vs 专家**：新手转发新闻说"我们也要做"；专家分析适用性给出判断。

→ 对应：职责 6.2 / 能力 5.2 / 知识 3.30
